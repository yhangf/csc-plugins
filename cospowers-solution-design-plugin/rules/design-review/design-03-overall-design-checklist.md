# 模块 M3：总体设计

## 模块概述

**审查目的**：确保总体设计文档在架构分析、目标量化、模块划分、系统流程、模块职责和关键特性设计方面达到完整的质量标准。

**适用范围**：总体设计文档。

**说明**：本模块共 27 个检查项，分为 7 个类别。

---

## 00. 文档写作

### 0.1 文档写作要求

**检查内容**：遵守《文档写作规范》子表中的所有要求。

**评判标准**：参见 reference `01-doc-writing-standards.md` 中的全部检查项。此项为汇总引用，审查时请参照 M1 逐项检查。

## 01. 技术需求设计

### 1.1 架构需求分析

**检查内容**：
1. 涉及架构变动的需要填写此章节，包括但不限于：新产品、新架构、重大重构版本；涉及跨产线的架构融合项目
2. 要充分考虑长期潜在需求，当前版本所选型的方案要可以满足未来需求，或者能够低成本演进。内容已经通过架构师或 TL 的认可
3. 不涉及的明确说明不涉及

**评判标准**：
- 通过：架构需求分析全面，方案具备可演进性，有架构师/TL 认可
- 不通过：涉及架构变动但分析不足，未考虑长期演进，或缺少架构师/TL 认可
- 不适用：不涉及架构变动，且已明确说明

### 1.2 特性需求分析

**检查内容**：按附件文档要求填写，基于用户 & 系统需求进行场景识别，确保需求场景无遗漏。

**评判标准**：
- 通过：场景识别全面，覆盖用户和系统需求
- 不通过：需求场景有明显遗漏
- 不适用：按实际情况判断

### 1.3 产品预期目标

**检查内容**：用户视角、用户可感知的产品预期目标，需要考虑当前版本以及未来演进的目标，起到牵引作用。

**评判标准**：同 M2-1.3，目标是用户视角、可量化、区分当前和未来演进。

### 1.4 方案设计

**检查内容**：方案设计要基于用户 & 系统需求→场景→技术思路→方案选型的推导分析设计思路。

**评判标准**：
- 通过：推导路径完整（需求→场景→技术思路→方案选型），逻辑清晰
- 不通过：推导过程跳跃，或场景与方案之间缺乏关联

### 1.5 技术预研和中台复用

**检查内容**：按要求填写"产品技术风险消除 checklist"附件，技术预研项目有明确的结论。若是带风险通过，还需要在方案选型中体现具体的风险细节。有中台复用的结论，后续要有方案选型过程。

**评判标准**：同 M2-1.4。

### 1.6 方案选型

**检查内容**：
1. 要包含技术预研、中台复用的选型
2. 方案选型明确说明每个方案的优缺点，要基于业务识别出来的关键技术维度进行评估，包括但不限于功能、性能、可测试性、可扩展性。最终采用的是哪个方案，选择该方案的具体原因。选择该方案必须得到架构师确认（上 TMG 评审的，需通过 TMG 认可）

**评判标准**：同 M2-1.5。

## 02. 设计目标

### 2.1 设计目标细化

**检查内容**：
1. 基于"产品预期目标"进一步细化各设计指标，包括用户可感知和不可感知的设计目标
2. 每个维度的设计目标设定要有依据支撑，"总设设计目标分析"附件需填写完整
3. 不涉及的明确说明不涉及，内容已经通过架构师或 TL 的认可

**评判标准**：同 M2-2.1。

### 2.2 中台复用需求

**检查内容**：如存在可复用中台，优先考虑使用中台提升研发效率。需列出关键的中台技术要求，并且与中台负责人确认。

**评判标准**：同 M2-2.2。

### 2.3 托管云需求明确

**检查内容**：明确产品是否要上托管云。在上托管云的前提下，已经和托管云接口人对接并确认上托管云是否有需要改造或调整的点。

**评判标准**：
- 通过：明确了是否需要上托管云，如果需要，已与接口人对接并确认改造点
- 不通过：未明确是否上托管云，或明确要上但未与接口人对接
- 不适用：确认不上托管云，已明确说明

## 03. 软件总体架构

### 3.1 模块结构图

**检查内容**：有完整的总体结构图表和对该图表的完整描述。允许多张图表示，但该版本涉及的任意一个模块必须体现在某一张总体结构图中。建议采用 4+1 模式来描述。

**评判标准**：
- 通过：有模块结构图且覆盖该版本所有涉及的模块，有对应的文字描述
- 不通过：无模块结构图，或图表未覆盖版本中涉及的模块

### 3.2 模块列表描述

**检查内容**：明确描述该版本所有变更模块（新增、修改、删除）的名称、功能以及变动内容。

**评判标准**：
- 通过：有清晰的模块列表，各模块的名称、功能和变更内容明确
- 不通过：模块列表不完整，或功能和变更内容描述模糊

### 3.3 需求覆盖完整

**检查内容**：所有新增和改动的模块能和需求点一一对应起来，模块设计能覆盖系统需求中的所有功能点。如果涉及跨平台需求，需要描述本设计覆盖的平台承接了哪些需求。

**评判标准**：
- 通过：模块与需求形成一一映射，所有功能点都有模块承接
- 不通过：存在需求点没有对应模块承接，或跨平台需求未做平台覆盖说明

### 3.4 模块设计策略

**检查内容**：明确本次新增或改动的模块需要执行哪个设计策略（概要设计/微型设计/不做设计）。

**评判标准**：
- 通过：每个新增/改动模块都有明确的设计策略
- 不通过：存在模块未指明设计策略

### 3.5 性能指标

**检查内容**：在模块设计要求小节中有明确说明每个模块需要达到的性能指标及资源约束（即该模块可以用多少内存、CPU、磁盘、描述符、网络带宽）。

**评判标准**：
- 通过：每个模块有量化的性能指标和资源约束
- 不通过：性能指标缺失或未量化（如"性能要足够好"）

**通过示例**：
> "传输模块：内存上限 512MB，CPU 使用率不超过 30%（单核），最大并发连接数 10000"

**不通过示例**：
> "传输模块应保证较好的性能"（未量化）

## 04. 系统流程

### 4.1 流程覆盖

**检查内容**：有描述所有涉及到多个模块的业务流程。敏捷开发中仅要求描述主要流程（即实现主要价值点的流程）。

**评判标准**：
- 通过：所有跨模块的业务流程都有描述
- 不通过：存在跨模块流程未描述，或敏捷模式下的主要流程缺失

### 4.2 流程设计

**检查内容**：文档中描述到的所有流程，不得以代码方式描述。建议以流程图、时序图、活动图、数据流图或伪码方式描述。

**评判标准**：
- 通过：所有流程用图表或伪码方式描述，无代码片段
- 不通过：存在用代码方式描述的流程

### 4.3 系统启动顺序

**检查内容**：总体设计需明确描述各模块的启动（或加载）顺序，及相关的依赖性。

**评判标准**：
- 通过：有明确的启动顺序描述，依赖关系清晰
- 不通过：缺少启动顺序描述，或依赖关系不清晰

### 4.4 可追溯性

**检查内容**：
1. 流程设计中涉及到的模块都应该在系统结构设计中有明确定义
2. 流程设计中描述的流程，必须可以追溯到需求文档中的某个需求或者总设任务书中的某个目标，不能无中生有

**评判标准**：同 M1-3.1。

## 05. 模块设计

### 5.1 模块职责明确

**检查内容**：有描述本模块的职责，在后续功能扩展时作为新增能力是否应该添加到本模块的依据。

**评判标准**：
- 通过：模块职责定义清晰，可作为功能归属的判断依据
- 不通过：模块职责描述模糊，无法判断新功能是否应归属该模块

**通过示例**：
> "传输模块职责：负责分支与总部之间的可靠数据通道，包括连接建立、保持和释放。评估标准：任何涉及跨节点数据搬运的功能应归入本模块，数据处理类功能不应归入本模块。"

**不通过示例**：
> "传输模块：负责数据传输相关的功能"（职责边界模糊）

### 5.2 设计约束描述清晰

**检查内容**：明确本模块有哪些设计约束，比如不能用套接字通信、不能用全局变量、需要考虑多实例等。

**评判标准**：
- 通过：设计约束列举清晰完整
- 不通过：存在已知约束未列出，或约束描述模糊

### 5.3 模块交互边界明确

**检查内容**：明确本模块与其他模块的交互途径，未描述的交互途径不得使用。比如明确只能用 RESTful API 交互，就不能通过文件等途径交互。

**评判标准**：
- 通过：交互途径明确列举，边界清晰
- 不通过：交互途径未列明，或描述模糊可能导致越界交互

## 06. 关键特性设计

### 6.1 维护设计

**检查内容**：有描述为了系统维护方便而在程序中做了哪些设计，比如异常告警、后台交互式调试、动态 debug 日志等。

**评判标准**：
- 通过：有具体的可维护性设计描述
- 不通过：无任何可维护性设计考虑
- 不适用：文档类型不需要此内容

### 6.2 安全设计要求

**检查内容**：执行威胁建模，并根据威胁建模的分析结果，对安全风险设计对应的应对机制。如果系统没有安全风险，可跳过。如经评估无执行必要，在通过安全经理认可后，可以裁剪掉本章节内容。

**评判标准**：
- 通过：有威胁建模分析和对应的安全机制设计，或有安全经理认可的免检说明
- 不通过：有安全风险但未做威胁建模或未设计应对机制
- 不适用：经安全经理认可无需执行，已明确说明

### 6.3 可靠性设计

**检查内容**：定义可靠性目标，给出系统冗余设计实现框架，给出故障检测与自愈整体框架，包括故障检测与自愈的层级、各个层级的功能以及层级之间的交互关系。需遵循不贰过原则。

**评判标准**：
- 通过：有可靠性目标、冗余设计、故障检测与自愈框架，遵循不贰过原则
- 不通过：无可靠性目标或缺失故障自愈机制，未提及不贰过原则
- 不适用：文档类型不需要此内容，已明确说明

### 6.4 可见性设计

**检查内容**：描述系统需要设计哪些对应的机制，来达成总设任务书中的可见性要求。平台类设计必须填写，不能跳过。

**评判标准**：
- 通过：有对应的可见性机制设计
- 不通过：平台类设计缺少可见性设计，或可见性设计无法满足任务书要求
- 不适用：非平台类设计

### 6.5 可重用设计

**检查内容**：明确说明本版本能复用哪些公共模块，能产出哪些公共模块。

**评判标准**：
- 通过：有明确的复用和产出说明
- 不通过：可复用的公共模块未提及，或可产出的公共模块未规划
