# 模块 M4：概要设计

## 模块概述

**审查目的**：确保概要设计文档在需求覆盖、方案选型、结构设计、概要流程、数据结构、流程设计、函数定义、OpenAPI 规范和数据规范方面达到完整的质量标准。

**适用范围**：概要设计文档。

**说明**：本模块共 27 个检查项，分为 10 个类别。

---

## 00. 文档写作

### 0.1 文档写作要求

**检查内容**：遵守《文档写作规范》子表中的所有要求。

**评判标准**：参见 reference `01-doc-writing-standards.md` 中的全部检查项。

## 01. 需求、场景及工作原理

### 1.1 需求覆盖

**检查内容**：覆盖了系统需求及总体设计中规定的由本模块实现的所有功能、特性和接口。即：需求中每一个功能点都有明确说明怎么实现，总设规定本模块实现的每一个接口都有明确说明怎么实现。必须满足需求/总设规定的各种约束，比如性能指标、资源使用限制。

**评判标准**：
- 通过：所有需求点的实现方案明确，所有接口实现说明清晰，性能/资源约束有回应
- 不通过：存在需求点/接口未覆盖实现方案，或未满足性能/资源约束

**通过示例**：
> "需求 REQ-001（文件上传）：通过 FileUploader 组件实现，支持分片上传，内存占用 ≤ 256MB，满足总设对传输模块的资源约束"

**不通过示例**：
> "文件上传功能由文件上传模块实现"（仅一句话带过，无分配方案、无约束回应）

### 1.2 对手分析

**检查内容**：
1. 如果竞争对手有类似功能，概要设计中须对竞争对手的功能实现进行分析，说明对方方案的优缺点及值得借鉴的地方
2. 如果业界其他产品中有类似功能或方案，须选择较为突出的进行分析，并说明借鉴的地方

> **原因**：他山之石，可以攻玉。通过分析竞争对手的功能，可以发现对手方案可供借鉴之处。

**评判标准**：
- 通过：有对手/业界类似功能分析，有优缺点总结和借鉴说明
- 不通过：有类似产品/功能但未做对手分析
- 不适用：该功能领域无已知竞争对手或类似方案，已明确说明

### 1.3 性能指标

**检查内容**：对本模块需要达到的性能指标有明确的定义，对如何达到性能指标有明确说明。

**评判标准**：
- 通过：性能指标量化明确，实现策略有说明
- 不通过：性能指标缺失、不量化，或无实现策略说明

## 02. 方案选型

### 2.1 核心需求覆盖

**检查内容**：方案选项必须描述到所有核心的需求点的实现思路。核心需求指模块最关键的几个大需求点，至少应该覆盖到需求矩阵中"子模块"这一列中描述到的内容。

**评判标准**：
- 通过：所有核心需求点有对应的实现思路
- 不通过：核心需求点的实现思路缺失

### 2.2 方案选型

**检查内容**：明确说明每个方案的优缺点，包括功能、性能、可测试性、可扩展性。需明确说明最终采用的是哪个方案，选择该方案的具体原因。选择该方案必须得到架构师或 TL 的确认。

**评判标准**：
- 通过：方案对比维度完整（功能/性能/可测/可扩展），选择理由充分，有架构师/TL 确认
- 不通过：单一方案无对比、维度不完整、无架构师/TL 确认

### 2.3 风险控制

**检查内容**：对已知技术风险有技术预研报告或风险避免说明。需列明各个方案中存在的风险点，对每个风险点必须有技术预研结果或风险规避说明。

**评判标准**：
- 通过：所有风险点都有预研结果或规避说明
- 不通过：已知风险未列明，或缺少预研/规避说明

---

## 03. 结构设计

### 3.1 模块组成结构

**检查内容**：有结构图表（或类图）描述了该模块由哪些软件单元组成。该图表必须覆盖本模块中的所有跨编译单元使用的类及子模块。如果一张图表无法容纳所有软件单元，可以分解为多个子图表。

**评判标准**：
- 通过：结构图覆盖所有跨编译单元的类和子模块
- 不通过：结构图缺失，或未覆盖关键的软件单元

### 3.2 关系描述

**检查内容**：有结构图表（或类图）说明软件单元之间的关系，比如继承或引用关系。

**评判标准**：
- 通过：单元之间的关系（继承/引用/依赖等）描述清晰
- 不通过：有关系但未在结构图中体现

---

## 04. 概要流程

### 4.1 流程覆盖

**检查内容**：对于设计任务书中描述的需求点，在流程设计中都有体现。

**评判标准**：
- 通过：所有需求点在流程设计中可找到对应的流程
- 不通过：存在需求点无对应流程

### 4.2 可调试性

**检查内容**：概要设计中应分析模块在客户场景下的典型故障类型，针对这些典型故障类型说明调试、定位方案。

**评判标准**：
- 通过：典型故障类型有列举，调试定位方案明确
- 不通过：无故障类型分析，或无调试定位方案

**通过示例**：
> "典型故障：文件传输中断。定位方案：通过日志记录传输会话 ID 和已传输分片 ID，配合传输进度 API 可快速判断断点位置"

**不通过示例**：
> "遇到故障时通过日志排查"（过于笼统，无具体故障类型和定位方案）

### 4.3 可测试性

**检查内容**：概要设计中应描述该模块的测试方案，并明确说明设计上怎么保证该测试方案的可实现性。测试方案应保证该模块可独立测试，如无法做到需得到 TL 或架构师的认可。

**评判标准**：
- 通过：有测试方案，可独立测试或有 TL/架构师认可的不可独立测试说明
- 不通过：无测试方案，或测试方案明显不可行

### 4.4 安全性设计

**检查内容**：威胁建模分析完整，并且通过安全经理的评审认可。

**评判标准**：
- 通过：威胁建模完整，有安全经理认可记录
- 不通过：有安全风险但威胁建模不完整或未经安全经理认可
- 不适用：无安全风险，已明确说明

### 4.5 可靠性设计

**检查内容**：有对可能的故障点进行列举描述，同时针对故障有对应的处理措施。需遵循不贰过原则。

**评判标准**：
- 通过：故障点列举充分，处理措施明确，遵循不贰过原则
- 不通过：故障点分析缺失或处理措施缺失

### 4.6 避免业务长时间中断处理

**检查内容**：对于处理用户业务数据流的守护程序、驱动，必须描述所有可能导致用户业务长时间中断的故障可能，并给出合理的解决方法或规避措施。可能导致故障的原因至少包含：
1. 程序崩溃
2. 死锁
3. 死循环

**评判标准**：
- 通过：至少涵盖崩溃/死锁/死循环三类故障，有对应的处理措施
- 不通过：缺失其中某类故障分析，或无处理措施
- 不适用：不涉及守护程序或驱动

---

## 05. 数据结构设计

### 5.1 数据结构

**检查内容**：以图表的方式描述了本模块用到的所有全局数据结构（超过一个软件单元使用）及配置文件、数据库。

**评判标准**：
- 通过：所有全局数据结构有图表描述，包括配置文件和数据库
- 不通过：存在全局数据结构未描述，或描述不完整
- 不适用：无全局数据结构，已明确说明

---

## 06. 流程设计

### 6.1 关键算法

**检查内容**：设计中描述到的有性能（或效果）要求的算法，须说明该算法最常见的场景及性能最差的场景，并说明在这两个场景下可达到的性能及资源需求。

**评判标准**：
- 通过：对性能敏感算法有最佳/最差场景分析和性能资源说明
- 不通过：有性能要求的算法但未做场景分析
- 不适用：无性能敏感的算法

### 6.2 启动和退出流程

**检查内容**：有描述模块的启动和退出流程。启动流程需描述到所有全局性对象的初始化、各线程的启动顺序（如有多线程）。退出流程须描述到所有全局性对象的清理、各线程的退出顺序及方式（如有多线程）。

**评判标准**：
- 通过：启动和退出流程描述完整，覆盖全局对象和线程
- 不通过：缺少启动或退出流程，或描述不完整

### 6.3 模块可追溯

**检查内容**：
1. 流程设计中涉及到的模块都应该在系统结构设计中有明确定义
2. 流程设计中描述的流程，必须可以追溯到设计任务书中的某个目标（需求或模块整体目标），不能无中生有

**评判标准**：同 M1-3.1。

## 07. 函数定义

### 7.1 接口覆盖

**检查内容**：明确描述了所有软件单元对外（其他单元或其他模块）提供的所有接口。如果是类，指的是类的公有和保护函数。

**评判标准**：
- 通过：所有对外接口有完整描述
- 不通过：存在对外接口未描述

### 7.2 函数描述

**检查内容**：对每一个函数需描述：名称、功能、参数（包括类型、作用、取值范围）、返回值（包括类型、作用、取值范围）、以及该函数的其他前后置约束条件（比如需要访问的全局变量）。参数和返回值如果涉及到动态资源分配，应描述负责资源释放的责任者。

**评判标准**：
- 通过：每个函数描述完整（名称/功能/参数/返回值/约束），资源释放责任明确
- 不通过：存在函数描述不完整，或资源释放责任不明确

**通过示例**：
> ```
> 函数名：file_transfer_init
> 功能：初始化传输会话
> 参数：session_id（uint32_t，1~65535，传输会话 ID）
>       config（const TransferConfig*，配置指针，调用方负责生命周期）
> 返回值：0 成功，-1 参数无效，-2 资源不足
> 前条件：config 已初始化
> 后条件：传输线程启动
> 资源释放：session_id 对应的资源由调用方调用 file_transfer_cleanup 释放
> ```

**不通过示例**：
> "file_transfer_init：初始化传输功能"（缺少参数、返回值、约束和资源释放说明）

### 7.3 函数定义可追溯

**检查内容**：函数列表中描述到的函数，必须可以追溯到流程设计（或关键算法）中的某个步骤，或者明确说明为什么会有这个函数（比如为扩展保留的）。

**评判标准**：
- 通过：每个函数可追溯到流程或算法，或有无中生有的函数但有明确扩展说明
- 不通过：存在无法追溯且无说明的函数

## 08. OpenAPI 规范

> 如果本文档不包含 OpenAPI 改动，本节可以不检查，标记为"(Non)未使用"。

### 8.1 OpenAPI 范围定义

**检查内容**：识别以下四类 API 是否涉及：
1. 对公司其他产品线开放的 API（如 AF 和 EDR 提供给 SIP/XDR/MSS 联动的 API）
2. 对第三方开放的 API（如 AC/SASE 开放给用户用于实现功能定制的 API）
3. 对公网开放的 API（如云脑威胁情报以及云查杀相关 API）
4. 对合作伙伴开放的 API（如 XDR 合作伙伴生态建设）

**评判标准**：
- 通过：涉及 OpenAPI 时，范围定义明确
- 不通过：涉及 OpenAPI 但未做范围定义
- 不适用：(Non)本文档不包含 OpenAPI 改动

### 8.2 OpenAPI 规范

**检查内容**：相关模块设计涉及 OpenAPI，需遵循对应的 API 规范（RESTful API 格式、认证与 SSO、接口描述、兼容性）。

**评判标准**：
- 通过：涉及 OpenAPI 时，遵循了相关规范
- 不通过：涉及 OpenAPI 但未遵循规范
- 不适用：(Non)本文档不包含 OpenAPI 改动

---

## 09. 数据规范

### 9.1 遵从最新规范

**检查内容**：相关模块的设计需参照最新公告的数据规范进行设计，如组件的采集、上报模块。所有需要落库的数据结构都必须符合最新公告的数据规范。如果数据结构无需符合已有数据规范，需要由数据 TMG、CTO 共同确认后备案并留痕。

**评判标准**：
- 通过：数据相关设计遵循最新数据规范，或有 TMG/CTO 确认的例外备案
- 不通过：涉及数据存储但未提及数据规范合规性
- 不适用：不涉及数据落库

### 9.2 规范变更

**检查内容**：如当前数据规范不满足业务或技术需求需要变更的，请按照数据规范制定流程变更规范，不得私自变更。

**评判标准**：
- 通过：无变更需求，或变更已按流程进行
- 不通过：存在私自变更数据规范的情况
- 不适用：数据规范满足需求，无变更

### 9.3 兼容性

**检查内容**：需考虑向后兼容，避免造成历史系统不可用。如规范中某字段删除后，实现时需考虑是逻辑删除还是硬删除。

**评判标准**：
- 通过：有明确的向后兼容策略说明
- 不通过：涉及数据变更但未考虑兼容性
- 不适用：全新模块首次设计
