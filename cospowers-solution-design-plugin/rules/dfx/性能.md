# 性能规范

## 用途

本文档供 AI 执行两类任务：**需求设计**（新接口/新特性设计时对照性能基线）、**代码 Review**（评估实现是否满足性能要求）。

违规报告格式：`[规则ID] <接口路径> — <描述>`

强制等级：`[强制]` 必须遵守 / `[建议]` 推荐遵守

---

## DFX 性能基线等级定义

> 来源：TR1 DFX 需求基线。L0 = 底线（必须达成，不满足须有 Story 跟踪）、L1 = 演进（按业务选择）、L2 = 拔高（追求卓越）。

| 等级 | 含义 | 要求 |
|---|---|---|
| L0 | 底线 | 必须达成，不满足须有 Story 跟踪 |
| L1 | 演进 | 按业务需要选择性达成 |
| L2 | 拔高 | 追求卓越，可选 |

## 性能基线决策树

```
这是哪类模块？
  SCC 新模块（新开发的接口/页面）
    → 查询接口并发 1：≤100ms；并发 10：≤100ms；并发 40：≤100ms
    → 配置接口并发 1：≤100ms；并发  5：≤100ms；并发 10：≤100ms
  SCC 老模块（已有接口改动）
    → 查询接口并发 1：≤1.5s；并发 10：≤1.5s；并发 40：≤1s
    → 配置接口并发 1：≤1.5s；并发  5：≤1.5s；并发 10：≤1s
  SCP 新模块（新开发的接口）
    → 查询接口并发 1：≤100ms；并发 10：≤100ms
    → 配置接口并发 1：≤100ms；并发  5：≤100ms
  SCP 老模块（已有接口改动）
    → 查询接口并发 1：≤1.5s；并发 10：≤1s
    → 配置接口并发 1：≤1.5s；并发  5：≤1s
  大批量接口（批量创建/删除 ≥100 个）
    → 不做响应时间硬性要求，保证成功，耗时基本可接受即可
```

## 规则详情

### 性能基线

**PERF-LINE-01** `[强制]` 新模块接口（SCC/SCP）在单并发下查询和配置接口响应时间必须 ≤ 100ms；超出须上 TD 并拉通干系人评估。

违规信号：新模块接口单并发响应超过 100ms；未上 TD 记录超标问题。

```
测试背景：调研生产环境，预估可能的数据规模，按最大规模构造测试数据

SCC/SCP 新模块验收标准：
  查询接口（并发 1/10/40）：≤ 100ms
  配置接口（并发 1/5/10）：≤ 100ms
  同时关注 CPU/内存等资源消耗
  不合格上 TD
```

---

**PERF-LINE-02** `[强制]` 老模块接口（SCC/SCP）在单并发下响应时间必须 ≤ 1.5s，高并发（查询 40/配置 10）下必须 ≤ 1s；超出须上 TD。

违规信号：老模块接口改动后响应退化至 1.5s 以上；未做基线对比测试。

```
SCC/SCP 老模块验收标准：
  查询接口（并发 1/10）：≤ 1.5s；并发 40：≤ 1s
  配置接口（并发 1/5）：≤ 1.5s；并发 10：≤ 1s
  同时关注 CPU/内存等资源消耗
  不合格上 TD
```

---

**PERF-LINE-03** `[强制]` 大批量接口（批量操作 ≥100 个资源）不做响应时间硬性要求，但必须保证操作成功且耗时在用户可接受范围内；如不满足需拉通干系人评估。

违规信号：批量接口超时失败（非超时无响应问题，而是业务失败）；用户等待超过合理预期未给出进度反馈。

```
大批量接口（如批量创建/删除 100 个云主机）：
  ✅ 不做响应时间硬性要求（例如批量创建 100 个，耗时 10s 可接受）
  ✅ 必须保证操作最终成功完成
  ✅ 耗时超长时需给出进度反馈（如 ulog 进度）
  ❌ 操作失败/超时报错不可接受
```

---

### 特性性能要求

**PERF-FEAT-01** `[强制]` 新增特性上线前必须进行性能分析，明确关键场景的数据规模和响应要求；不满足基线的接口必须提前识别并评估。

违规信号：新特性无性能分析文档；特性上线后才发现性能问题。

关键场景性能参考：

| 特性 | 数据规模 | UI 响应 | API 响应 |
|------|---------|---------|---------|
| 物理主机 | 52 个物理机，含 40 个云主机及一年监控数据 | ≤ 1.8s | ≤ 300ms |
| 资源池 | 128 个资源池，6 万云主机 | ≤ 1.8s | — |
| 云主机&快照 | 6 万云主机，单集群 360 个快照 | ≤ 1.8s | — |

---

**PERF-FEAT-02** `[建议]` 新接口设计阶段应评估是否存在以下高风险场景，提前做好分页/异步/缓存设计：

| 高风险场景 | 建议方案 |
|---------|---------|
| 列表接口无分页，数据量可能超万 | 强制分页，设置 limit 上限（参考 GEN-RES-02） |
| 配置接口涉及跨服务级联调用 | 评估链路耗时，超 500ms 考虑异步化 |
| 查询接口无缓存，每次全量计算 | 热点数据加缓存，设置合理 TTL |
| 并发写操作可能产生锁竞争 | 评估数据库锁粒度，必要时排队或限流 |

---

### 测试数据规模

**PERF-DATA-01** `[强制]` 性能测试必须以生产环境最大数据规模为背景构造测试数据，不得用小数据量代替大数据量场景；测试前须调研生产环境实际数据规模。

违规信号：用 100 条数据测试声称支持 6 万云主机的接口；测试数据与生产规模差距超 10 倍。

```
各场景测试数据规模参考：
  物理主机：≥52 个物理机，含 40 个云主机，一年监控数据
  资源池：≥128 个资源池，6 万云主机背景
  云主机&快照：6 万云主机，单集群 360 个快照
  租户/用户：32 个租户，64 个用户，多个并发用户
```

---

### 稳定性

**PERF-STAB-01** `[建议]` 预发布前需长稳运行 7 天，且无 P1/P2 稳定性问题；正式发布前升级预发布包后再稳运行 7 天。

监控指标：CPU 使用率、内存使用率、操作日志异常、服务状态（restart/crash）。

### Web 控制台页面性能（TR1 DFX 用户侧基线 L0）

**PERF-WEB-01** `[强制]` Web 控制台页面响应性能：
- 高频页面响应流畅（<=3 秒）占比 >= 99%，不出现难以忍受（>=7 秒）的页面
- 低频页面响应流畅（<=3 秒）占比 >= 90%，难以忍受（>=7 秒）的页面 <= 1%
- 任何情况下不允许出现打开失败的页面

适用范围：所有具有 Web 控制台的产品。

---

### 业务可用性指标（TR1 DFX 系统侧基线 L0）

**PERF-BIZ-01** `[强制]` 业务可用性指标：
- 丝滑（99.99%~100%）占比 >= 90%
- 正常（99%~99.99%）占比 <= 10%
- 难以忍受（95%~99%）占比 0%
- 业务响应时间 <1 秒占比 >= 90%

适用范围：所有产品。

---

### 共存影响指标（TR1 DFX 部署侧基线 L0）

**PERF-COEX-01** `[强制]` 共存影响指标：
- 对客户现有业务系统的性能劣化 <5%，覆盖率 >= 90%

适用范围：与客户系统共存部署的产品。

## Code Review Checklist

逐项扫描，违规输出 `[规则ID] <接口路径> — <描述>`。

**性能基线**
- [ ] PERF-LINE-01 — SCC/SCP 新模块接口单并发是否 ≤ 100ms；超标是否上 TD
- [ ] PERF-LINE-02 — SCC/SCP 老模块接口改动后是否仍满足 ≤ 1.5s（单并发）/ ≤ 1s（高并发）
- [ ] PERF-LINE-03 — 大批量接口是否保证最终成功；超长耗时是否有进度反馈

**特性性能**
- [ ] PERF-FEAT-01 — 新特性是否有性能分析，明确数据规模和响应要求
- [ ] PERF-FEAT-02 — 高风险场景（大列表/跨服务链路/热点查询）是否有对应设计（建议）

**测试数据**
- [ ] PERF-DATA-01 — 性能测试是否以生产最大规模构造测试数据

**稳定性**
- [ ] PERF-STAB-01 — 预发布是否长稳运行 7 天，无 P1/P2 问题（建议）

**Web 控制台性能（TR1 基线 L0）**
- [ ] PERF-WEB-01 — 高频页面 <=3s 占比是否 >= 99%；低频页面 <=3s 占比是否 >= 90%；是否有打开失败页面

**业务可用性（TR1 基线 L0）**
- [ ] PERF-BIZ-01 — 99.99%~100% 占比是否 >= 90%；业务响应 <1s 占比是否 >= 90%

**共存影响（TR1 基线 L0）**
- [ ] PERF-COEX-01 — 对客户现有系统性能劣化是否 <5%；覆盖率是否 >= 90%
