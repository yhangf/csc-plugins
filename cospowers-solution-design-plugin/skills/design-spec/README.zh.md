# Design Specification — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

将系统需求转化为严格遵循团队标准模板的正式设计文档。**两种文档按顺序产出**：

1. **系统级设计文档**（总体设计，`system-design-template.md`，主入口 + 7 子文档）— 系统架构、子系统划分、跨子系统交互、系统级 DFX
2. **接口文档**（`openapi-template.yaml`）— 基于系统级设计定义子系统间的 API 契约、消息 Schema、事件定义

**产出顺序很重要：** 系统级设计定义子系统划分和交互 → 接口文档锁定契约 → 子系统设计（由 `subsystem-design-spec` skill 负责）在契约基础上实现内部细节。

**核心原则：** 无空章节、必填 Mermaid 图、DFX 全覆盖、需求可追溯、评审就绪。

## 14 步检查清单

1. **加载系统需求** — 读取需求文档，理解所有 REQ 条目
2. **检查历史设计文档** — 如存在旧版本，记录已有哪些子系统/接口/流程
3. **查询产品知识库** — 企业项目调用 `kb-query`（关键词全部用中文）；**无 kb-query 则降级到本地 `doc/kb/`，再无则直接代码优先（Glob/Grep）**
4. **搜索知识库文档** — 调用 `daedalus-knowledge`；**不可达则跳过，使用本地 `rules/` 规范文件**
5. **评估复杂度并读取模板** — 加载 `templates/system-design-template.md`（主入口 + 7 子文档）
6. **收集系统设计约束** — 通过 init 模板收集架构背景情报，含 HARD GATE 完整性校验和知识归档
7. **呈现设计方案** — 2-3 个方案 + **Challenger 模式**：对每个方案问"什么会失败"，标注 `[待确认]`
8. **编写系统级设计文档** — 按 7 个子文档逐步写，逐检查点确认：
   - 子文档 1: 技术需求设计（架构TRD / 特性TRD / 需求追溯）
   - 子文档 2: 设计目标（性能/资源/可靠性等 12 维量化目标）
   - 子文档 3: 硬件方案
   - 子文档 4: 软件方案（4+1 视图、子系统划分⭐、方案选型、交互流程⭐、数据架构⭐）
   - 子文档 5: 关键特性设计（安全/可靠性/性能/可测试性/可运维性/部署⭐/兼容性/风险分析）
   - 子文档 6: 总结（版本变化、风险保证、尚未解决的问题）
   - 子文档 7: 变更控制（变更列表、修订记录）
9. **编写接口文档** — 以 **OpenAPI 3.0 YAML** 格式输出，字段约束使用 sf-apidesc 类型映射
10. **跨章节一致性检查** — DFX 数字对齐、需求追溯完整、术语一致
11. **子代理评审** — 派遣独立子代理使用 `design-document-reviewer-prompt.md` 评审设计文档
12. **质量评估（关卡）** — 派遣 `sysdesign-evaluator` 无上下文子代理（`Agent(subagent_type="general-purpose")`），M1-M6 模块化审查 + 3 条红线 + 加权打分，必须达到 B 级（≥ 80 分）
13. **用户审查**
14. **转入子系统设计** — 调用 `subsystem-design-spec` 编写各子系统实现设计

## M1-M6 审查模块

系统设计文档评估依据 6 个审查模块，按设计子类型组合：

| 模块 | 全称 | 检查内容 | 总体设计 | 概要设计 | 技术需求设计 |
|------|------|---------|:--:|:--:|:--:|
| M1 | 文档书写规范 | 格式、术语、图表、编号、引用 | ✅ | ✅ | ✅ |
| M2 | 技术需求设计 | 需求追溯、设计与需求映射、技术选型依据 | ✅ | — | ✅ |
| M3 | 总体设计 | 架构方案、模块划分、技术栈、系统流程 | ✅ | — | — |
| M4 | 概要设计 | 函数定义、接口设计、数据结构、模块内部流程 | — | ✅ | — |
| M5 | 国际化i18n设计 | 多语言支持、时区处理、字符编码 | ✅ | ✅ | ✅ |
| M6 | 不贰过设计准则 | 历史故障经验、已知陷阱规避 | ✅ | ✅ | ✅ |

## 逐层对齐（步骤 8 检查点）

设计文档**分层写、逐检查点确认**，每个检查点是天然决策边界：

| 检查点 | 子文档 | 确认内容 |
|-------|--------|---------|
| 技术需求 | 子文档 1 | 架构需求分析完整？特性需求覆盖？ |
| 软件方案 | 子文档 4 | 子系统划分对？核心流程合理？数据架构正确？ |
| DFX | 子文档 5 | 性能目标现实？安全方案通过？可靠性架构合理？ |
| 总结 | 子文档 6 | 所有遗留问题已标注？后续工作明确？ |

## 三种 AI 角色模式

| 模式 | 何时使用 | 核心动作 |
|------|---------|---------|
| **Writer（写作者）** | 生成文档内容 | 按模板逐章填写 |
| **Challenger（挑战者）** | 方案选型和写作全程 | 问"什么会失败"、"我在假设什么"、"有没有更简单的方案" |
| **Reviewer（评审者）** | 子代理评审 + sysdesign-evaluator | 逐条扫描打分，M1-M6 模块化审查 |

## 复杂度路由

| 场景 | 产出 |
|------|------|
| 单子系统/模块变更，无跨系统决策 | 直接使用 `subsystem-design-spec`（主入口 + 8 子文档），接口文档可选 |
| 多子系统影响、架构决策、跨系统依赖 | 本 skill 产出：系统级设计 → 接口文档，然后转 `subsystem-design-spec` |

## 3 条设计红线

| 红线 | 要求 |
|------|------|
| **章节完整性** | 所有子文档章节不得为空，缺失 ≥ 3 → 红线违反 |
| **可靠性章节** | 必须覆盖：载体组件容错 / 资源使用分析 / 业务流程可靠性 |
| **安全性章节** | 必须覆盖：安全基线 / 威胁建模 / 数据保护 |

## 下一步

系统设计完成并通过质量评估后，调用 `subsystem-design-spec` 编写子系统级实现设计文档。
