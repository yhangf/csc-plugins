# Subsystem Evaluator — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

对 `subsystem-design-spec` 产出的子系统设计文档（`*-<subsystem>-design.md`，排除 `*-system-design.md`）进行单文档逐条规范扫描和质量打分。加载概要设计 checklist 并应用 5 项子系统级裁剪适配，结合 M1/M4/M5/M6 模块和 FMEA 分析表专项检查。跨文档一致性由 `doc-quality-evaluator` 负责。

## 审查流程

```
Phase 0: 加载（自动发现文档 → 加载规范 + 5 项裁剪适配 + M1/M4/M5/M6 + FMEA 检查）
  → Phase 1: 逐条规范扫描（规范提取项 + 9 条 SUB- 内置项 + 3 条红线 + FMEA 专项）
  → Phase 2: 跳过（跨文档一致性由 doc-quality-evaluator 承担）
  → Phase 3: 加权打分
  → Phase 4: 生成评估报告
```

## 5 项子系统级裁剪适配

概要设计 checklist 在子系统级做以下适配：

| 原检查项 | 适配方式 |
|------|------|
| 项 1.2 对手分析 | 跳过（子系统级不做竞争对手分析） |
| 项 4.2 可调试性 | 映射到子文档 3 §3.7.4 可调试性设计 |
| 项 4.6 业务长时间中断处理 | 映射到子文档 3 §3.5 异常处理策略 |
| 项 6.2 启动和退出流程 | 跳过（子系统级不单独设计启动退出） |
| 项 9.1/9.2 数据规范 | 转化为"接口规范遵从"（对应子文档 2 对外接口） |

## M1/M4/M5/M6 模块标注

| 模块 | 全称 | 子系统级适用范围 |
|------|------|------|
| M1 | 文档书写规范 | 全文档适用 |
| M4 | 概要设计规范 | 对应子文档 2/3/4/5（对外接口/概要说明/数据结构/流程设计） |
| M5 | 国际化i18n设计 | 触发条件：子系统涉及用户可见字符串或时间显示 |
| M6 | 不贰过设计准则 | 全文档适用，特别是子文档 3（概要说明） |

## FMEA 分析表检查（§0.3）

在子文档 3 可靠性章节的业务流程可靠性部分，对 FMEA 分析表执行 6 维度专项检查：
- 功能分析完整性（功能点覆盖、功能要求可验证）
- 失效分析完整性（失效模式、失效影响、失效原因追溯根因）
- SOD 评分准确性（S/O/D 值域 1-10、评分合理性）
- AP 优先级正确性（来源基于 SOD、对照 AP 映射表验证）
- 改进措施有效性（针对失效原因、可量化验证、责任人明确）
- 改进落地跟踪（开发自检、测试用例覆盖、用例 ID 填写）

## 评分体系

| 维度 | 权重 | 说明 |
|------|:--:|------|
| 技术可行性 | 35% | 组件真实可用，逻辑无漏洞，方案设计合理 |
| 可测试性 | 25% | 功能流程覆盖正常/异常/边界三类场景 |
| 需求完整性 | 20% | 所有设计场景被覆盖，无遗漏功能点 |
| 规范符合性 | 15% | 文档格式、命名、章节完整性 |
| 架构合理性 | 5% | 设计模式与现有架构一致，边界清晰 |

扣分：Critical -40 / Error -20 / Warning -8。

## 3 条红线（一票否决）

| 红线 | 标准 |
|------|------|
| R1 章节完整性 | 缺失章节数 ≥ 3（主入口 + 8 子文档） |
| R2 可靠性章节 | 必须覆盖故障容错/资源使用/业务流程可靠性（含 FMEA 专项） |
| R3 安全性章节 | 必须覆盖安全基线/威胁建模/数据保护 |

## 派遣方式

由 `subsystem-design-spec` 步骤 3（每个子系统设计完成后）以隔离子代理方式派遣。评估完成后将评级、红线状态、问题表格、报告路径返回给调用方。

## 下一步

- 评级 A/B（≥ 80）→ `subsystem-design-spec` 继续下一步
- 评级 C/D/F → 修复问题后重新派遣 `subsystem-evaluator`，直至达到 B 级

> 跨文档一致性核查需运行 `doc-quality-evaluator` 全套评估。
