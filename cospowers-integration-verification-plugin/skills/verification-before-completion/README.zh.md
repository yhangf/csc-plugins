# Verification Before Completion — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

声称工作完成却未验证，不是高效——是不诚实。

**核心原则：** 证据先于声明，始终如此。

## 铁律

```
没有新鲜验证证据，不能声称完成
```

如果当前消息中没有运行验证命令，就不能说它通过了。

## 门控函数

在声称任何状态或表达满意之前：

1. **识别** — 什么命令能证明这个声明？
2. **运行** — 执行完整命令（新鲜、完整）
3. **阅读** — 完整输出，检查退出码，计算失败数
4. **验证** — 输出是否确认声明？
   - 否：陈述实际状态并附证据
   - 是：陈述声明并附证据
5. **然后才能** — 做出声明

跳过任何步骤 = 在说谎，不是在验证。

## 常见失败模式

| 声明 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 失败 | 之前的运行、"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查、推断 |
| 构建成功 | 构建命令：退出码 0 | Linter 通过、日志看起来正常 |
| Bug 已修复 | 原始症状测试：通过 | 代码改了、假设修好了 |
| 提交规范合规 | spec-commit 检查清单全过 | 只检查了编译 |
| 代理完成 | VCS diff 显示变更 | 代理报告"成功" |
| 需求满足 | 逐行对照清单 | 测试通过 |

## 红旗（立即停止）

- 使用"应该"、"可能"、"似乎"
- 验证前表达满意（"太好了！"、"完美！"、"搞定！"）
- 即将提交/推送/PR 但未验证
- 信任代理的成功报告
- **任何暗示成功但未运行验证的措辞**

## 合理化防护

| 借口 | 现实 |
|------|------|
| "现在应该行了" | 运行验证 |
| "我很有信心" | 信心不等于证据 |
| "就这一次" | 没有例外 |
| "Linter 过了" | Linter 不等于编译器 |
| "代理说成功了" | 独立验证 |

## 何时适用

**始终在以下操作之前：**
- 任何成功/完成声明
- 任何满意表达
- 提交、PR 创建、任务完成
- 进入下一个任务
- 委派给代理
- 执行 `git commit`

## 底线

**验证没有捷径。** 运行命令、阅读输出、然后才能声明结果。不可协商。
