# doc-quality-evaluator — 文档质量评估技能

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

对 agent-rules-plugin 工作流（`system-requirement-analysis` / `design-spec` / `roles.yaml` 角色）产生的文档进行结构化质量评估。采用**逐条规范扫描**（一条规则一条规则地实时检查）+ **跨文档一致性核查**（三份文档描述的方案必须自洽）+ **加权打分** + **定位到章节的详细问题报告**。

**评估顺序（不可跳步、不可乱序）：**

```
Phase 0: 加载 → Phase 1: 逐条规范扫描 → Phase 2: 跨文档一致性核查 → Phase 3: 打分 → Phase 4: 生成报告
```

## 支持的文档类型

| 文档 | 产出 Skill | 文件命名规律 |
|---|---|---|
| 系统需求文档 | `system-requirement-analysis` | `*-system-requirements.md` |
| 系统设计文档（总体设计） | `design-spec` | `*-system-design.md` |
| 子系统设计文档（详细设计） | `subsystem-design-spec` | `*-<subsystem>-design.md` |
| OpenAPI 接口规范 | `design-spec` | `*-openapi.yaml` |

## 评估模式

| 模式 | 输入 | 支持的检查 |
|---|---|---|
| **单文档** | 1 份文档 | 规范扫描 + 打分 |
| **双文档** | 2 份文档 | 规范扫描 + 部分一致性 + 打分 |
| **全套**（推荐） | 3-4 份文档 | 全部检查 |

单文档模式无法执行跨文档一致性核查，报告中会明确注明。

## Phase 0：加载阶段

### 文档自动发现

**不需要用户提供路径。** Skill 主动搜索，按优先级依次执行：

1. **标准目录**：`docs/agent-rules/specs/` 下按文件名模式匹配
2. **上下文路径**：用户对话中提到的目录
3. **向上递归**（兜底）：从当前目录向上最多 3 层搜索

搜索完成后，根据发现的文档总数走不同路径：

**🟢 单文档（总数 = 1）** — 直接开始评估，不询问用户，声明当前评估的文档类型和路径后进入 Phase 1。Phase 2 跨文档一致性核查自动跳过。

**🔵 多文档（总数 ≥ 2）** — 展示发现结果（带编号和修改时间），推荐最新一套组合，**等待用户确认**后再开始评估。

发现结果示例（多文档）：
```
🔍 文档发现结果:

📋 系统需求文档 (2 个):
  [1] docs/agent-rules/specs/2026-04-10-xxx-system-requirements.md  ← 最新
  [2] docs/agent-rules/specs/2026-03-01-xxx-system-requirements.md

🏗️ 系统设计文档 (1 个):
  [3] docs/agent-rules/specs/2026-04-10-xxx-system-design.md

...

💡 建议评估组合: [1] [3] [4] [5] [6]

请确认：A) 使用建议组合  B) 自定义编号  C) 手动提供路径
```

**⚠️ 多文档模式下必须等待用户确认才能进入 Phase 1。单文档模式直接开始。**

### 规范加载

根据输入文档类型读取对应规范文件（必须实际读取，不允许跳过）：

| 文档类型 | 需要读取的规范 |
|---|---|
| 系统需求文档 | 概要设计checklist、文档撰写规范、不贰过设计准则 |
| 系统设计文档 | 总体设计review规范、文档撰写规范、模块设计规范、不贰过设计准则、dfx/安全、dfx/性能 |
| 子系统设计文档 | 详细设计checklist、模块设计规范、文档撰写规范、不贰过设计准则、dfx/安全 |
| OpenAPI 规范 | RESTful_API格式规范v3.0、01-OPENAPI开发规范、API开发规范、错误码规范 |

加载完成后，从规范文件中提取所有检查项，构建带编号的检查清单（DOC-01、SYS-01、SUB-01、API-01…）。

## Phase 1：逐条规范扫描

### 实时输出格式

对每一条检查项，固定输出：

```
📍 检查 [CHECK-ID]: [规则摘要（来自规范文件的原始描述）]
  ├─ 检查位置: § X.X [章节名] 或 "全文"
  ├─ 引用原文: "[被检查内容的直接引用]"
  ├─ 规范要求: [该规则的具体要求]
  ├─ 检查结果: ✅ 通过 / ❌ 违规 / ⚠️ 警告
  └─ [仅违规时] 问题ID: ISSUE-XXX | 级别: Critical/Error/Warning | 扣分: -N分
```

通过的规则**不写入最终报告**（只在实时检测过程中输出）。

### 违规记录格式

```
[ISSUE-XXX]
  文档: [文档名]
  位置: § [章节号] [章节名]
  规则: [CHECK-ID] — [规则描述]
  引用原文: "[文档中的原文，精确到句子]"
  问题描述: [具体说明哪里违反了规则]
  严重级别: Critical / Error / Warning
  修改建议: [具体可操作的修改方向]
  扣分: [-N分]（维度: [维度名]）
```

### 扫描完成统计

每份文档扫描完毕后输出：

```
📊 [文档名] 扫描完成
  - 扫描规则数: N 条
  - 通过: n 条
  - 违规(Critical): n 条 | 违规(Error): n 条 | 警告(Warning): n 条
  - 发现问题: ISSUE-001 至 ISSUE-XXX
```

### 三条设计红线（一票否决）

以下三条红线优先检查，任一违反则最终总评直接为 **F**，不论其他维度得分：

| 红线 | 标准 | 记录内容 |
|---|---|---|
| **R1 章节完整性** | 缺失必要章节 ≥ 3 个 | 缺失了哪些章节（具体章节号和名称） |
| **R2 可靠性章节** | 必须存在且有实质内容（不能只写"不涉及"或"参见平台"） | 是否覆盖：故障容错分析 / 资源使用分析 / 业务流程可靠性 |
| **R3 安全性章节** | 必须存在且有实质内容 | 是否覆盖：安全基线 / 威胁建模 / 数据保护 |

## Phase 2：跨文档一致性核查

> 仅在提供 2 份或以上文档时执行。单文档模式跳过，在报告中注明。

### A. REQ 追踪（需求 → 设计）

逐条扫描系统需求文档中的每个 REQ-XXX，输出：

```
📍 追踪 REQ-[XXX]: [需求名称]
  ├─ 需求描述: [需求文档中的描述摘要]
  ├─ 设计文档 § 1.4 追踪表: ✅ 有记录 / ❌ 缺失
  ├─ 对应设计元素: [章节 § X.X] 或 "未找到对应设计"
  ├─ 方案一致性: ✅ 一致 / ❌ 不一致 / ⚠️ 部分覆盖
  └─ [不一致时] 具体差异: 需求说 "[...]"，设计做了 "[...]"
```

### B. 技术方案一致性（系统设计 → 子系统设计）

系统设计 § 4.1 选定的技术方案（技术栈、架构模式、数据库、中间件等）必须在子系统设计中得到体现：

```
📍 方案一致性检查: [方案名称/技术选型]
  ├─ 系统设计选型: "[系统设计§4.1中的选型结论]"
  ├─ 子系统设计体现: "[子系统设计中对应的描述]" / "未找到"
  ├─ 一致性: ✅ 一致 / ❌ 矛盾 / ⚠️ 子系统用了不同方案但未说明原因
  └─ [矛盾时] 具体差异: [...]
```

**常见方案漂移场景：**
- 系统设计选 Redis 缓存，子系统设计改用本地内存缓存，无说明
- 系统设计定义事务用 2PC，子系统设计改用 Saga，无说明
- 系统设计定义异步消息队列，子系统设计改为同步 RPC 调用，无说明

### C. API 契约一致性（OpenAPI → 子系统设计）

OpenAPI 规范中每个接口在子系统设计 § 3.1 的引用必须一致：

```
📍 接口契约检查: [HTTP方法] [路径]
  ├─ OpenAPI 规范定义: [method path, 关键参数/字段名/类型]
  ├─ 子系统设计引用: "[§3.1 中的引用内容]"
  ├─ 一致性: ✅ 一致 / ❌ 不一致
  └─ [不一致时] 差异: [字段名/类型/返回值不匹配]
```

### D. DFX 指标三份一致

系统需求 Ch.4（非功能性需求（DFX））的所有量化指标，必须与系统设计 DFX 章节、子系统设计 DFX 章节数字完全一致：

```
📍 DFX指标检查: [指标名称，如 P95响应时间]
  ├─ 系统需求 §4 值: [value + 单位]
  ├─ 系统设计 DFX 值: [value + 单位] / "未找到"
  ├─ 子系统设计 DFX 值: [value + 单位] / "未找到"
  ├─ 一致性: ✅ 三份一致 / ❌ 不一致
  └─ [不一致时] 具体数字差异: [...]
```

### E. 子系统边界一致性

系统设计中定义的子系统职责，在子系统设计 § 2 中必须匹配：

```
📍 子系统边界检查: [子系统名称]
  ├─ 系统设计中的职责定义: "[...]"
  ├─ 子系统设计 §2 中的职责描述: "[...]"
  ├─ 一致性: ✅ 一致 / ❌ 职责描述有矛盾 / ⚠️ 扩大/缩小了职责范围但未说明原因
  └─ [不一致时] 差异: [...]
```

## Phase 3：打分

### 评分维度与权重

| 维度 | 权重 | 说明 |
|---|---|---|
| **跨文档一致性** | 30% | 三份文档方案、数字、边界必须自洽 — 方案漂移是最危险的问题 |
| **需求完整性** | 20% | 所有 REQ 都有对应设计，所有场景都被覆盖 |
| **技术可行性** | 20% | 组件真实可用，逻辑无漏洞，性能无明显瓶颈 |
| **规范符合性** | 15% | 文档格式、命名、章节完整性等符合团队规范 |
| **架构合理性** | 10% | 设计模式与现有架构一致，依赖合理 |
| **可测试性** | 5% | 可注入依赖，有测试接口，支持隔离测试 |

### 扣分规则

| 级别 | 说明 | 每个问题扣分 |
|---|---|---|
| **Critical** | 红线违反、方案根本性矛盾、安全漏洞 | -40 分 |
| **Error** | 需求未覆盖、逻辑错误、数字不一致、空章节 | -20 分 |
| **Warning** | 命名不规范、建议性问题、数字无来源 | -8 分 |

每个维度最低 0 分（不出现负数）。严格扣分示意：1 个 Critical 使维度直接跌至 60 以下；2 个 Error 使维度跌至 60；3 个 Warning 扣 24 分。

总分 = 各维度得分 × 权重之和。

### 评级

| 总分 | 等级 | 含义 |
|---|---|---|
| 95–100 | **A** | 优秀，可直接进入实施规划 |
| 80–94 | **B** | 良好，必须修复所有 Error 级问题后才能进入实施 |
| 65–79 | **C** | 勉强，必须修复所有 Error+ 问题并重新评估 |
| < 65 | **D** | 不及格，大幅返工后重新评估 |
| 红线违反 | **F** | 直接失败，不论其他分数 |

### 打分输出样例

```
📊 评分明细:

┌─────────────────────┬──────┬──────────┬────────────────────────┐
│ 评分维度            │ 权重 │ 维度得分 │ 扣分明细               │
├─────────────────────┼──────┼──────────┼────────────────────────┤
│ 跨文档一致性        │ 30%  │  70/100  │ ISSUE-003(-10), ...    │
│ 需求完整性          │ 20%  │  75/100  │ ISSUE-007(-25)         │
│ 技术可行性          │ 20%  │  90/100  │ ISSUE-012(-10)         │
│ 规范符合性          │ 15%  │  85/100  │ ISSUE-015(-3), ...     │
│ 架构合理性          │ 10%  │ 100/100  │ -                      │
│ 可测试性            │  5%  │  97/100  │ ISSUE-019(-3)          │
├─────────────────────┼──────┼──────────┼────────────────────────┤
│ 加权总分            │ 100% │  80.25   │                        │
└─────────────────────┴──────┴──────────┴────────────────────────┘

红线状态: R1 ✅ | R2 ✅ | R3 ✅
总评: B — 80.25分，建议修复 Error 级问题后进入实施规划
```

## Phase 4：生成报告

报告保存至：`[文档所在目录]/quality-reports/YYYY-MM-DD-<文档名>-quality-report.md`

### 报告结构

1. **红线检查结果** — R1/R2/R3 状态，违反时列出具体缺失内容
2. **评分总览** — 加权得分表格（含扣分明细）
3. **跨文档一致性核查结果**
   - REQ 追踪覆盖率表（REQ-XXX | 设计章节 | 状态）
   - 技术方案一致性问题（仅列不一致项）
   - DFX 指标三方对比表
4. **问题清单**（按 Critical → Error → Warning 排序，每个问题含位置+原文+规则+建议）
5. **每份文档单独评分**
6. **改进建议**（按优先级排序，含具体操作步骤）
7. **下一步行动**

## 常见问题类型

| 问题类型 | 典型表现 | 级别 |
|---|---|---|
| 方案漂移 | 系统设计选了A方案，子系统设计用了B方案，无说明 | Critical |
| DFX 数字不一致 | 需求说 P95<200ms，设计说 P95<500ms | Error |
| REQ 未覆盖 | REQ-008 在设计追踪表中找不到 | Error |
| 未验证假设 | 设计假设"用户已登录"但需求未明确该前提 | Error |
| API 契约矛盾 | OpenAPI 返回 `order_id`，子系统设计说返回 `id` | Error |
| N+1 查询 | 列表接口在循环中逐条查询 DB | Error |
| 空 DFX 章节 | 安全章节只写"参见平台安全机制" | Error |
| 缺 Mermaid 图 | ⭐必填章节只有文字描述无图 | Error |
| 命名不一致 | 字段在不同文档里分别叫 `userId`、`user_id`、`uid` | Warning |
| 数字无来源 | 性能目标"TPS > 10000"无 ADR 或历史数据支撑 | Warning |

## 一问题一类问题原则

发现某类问题（如"DFX 数字不一致"）后，立刻扫描全文同类实例，一并记录。不可只记录第一个发现的问题，漏掉同一文档中的其他同类问题。

## 定位说明

`doc-quality-evaluator` 是**被动评估器**，不主动路由下一步流程。

评估完成后，将评级、分数、问题表格、报告路径返回给**调用方 skill**，由调用方决定后续行动：

| 调用方收到的评级 | 调用方的后续行动 |
|---|---|
| **A / B（≥ 80）** | 调用方继续其自身下一步（由调用方 skill 的 Checklist 决定，与本 skill 无关） |
| **C（65–79）** | 调用方修复所有 Error 及以上问题后，重新派遣 `doc-quality-evaluator`，循环直到 B+ |
| **D（< 65）** | 调用方需较大返工，回到自身更早的 Checklist 步骤 |
| **F（红线违反）** | 调用方需大幅返工后重新提交评估 |
