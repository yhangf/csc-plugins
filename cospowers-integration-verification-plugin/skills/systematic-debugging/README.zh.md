# Systematic Debugging — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

随机修复浪费时间、引入新 bug。快速补丁掩盖根本问题。没有数据的猜测导致错误结论。

**核心原则：** 先找根因，再修复。症状修复即为失败。

**证据原则：** 真实数据高于一切。缺少证据时，停下来问——不要猜。

## 铁律

```
没有根因调查，不能提修复方案
没有证据，不能下结论
```

## 步骤 0：搜索知识 Hub

调查**之前**，先调用 `evo-knowledge-wheel` 搜索团队知识 Hub。可能已有现成方案（Capsule）、类似模式（Gene）或多步修复方案（Recipe）。

## 证据优先级

```
真实数据（API 响应、日志、数据库查询、截图、core dump）
  > 代码分析（调用链追踪、静态分析）
  > 文档（设计文档、API 规格）
  > 经验猜测（模式识别）
```

需要真实数据但没有时，**停下来向用户索取**。不猜、不编造、不转述日志——必须原文引用。

## 四个阶段

### 阶段 1：问题定义和根因调查（只读，不改代码）

#### 5W1H 信息收集

| 维度 | 问题 | 获取方式 |
|------|------|---------|
| What | 故障现象 vs 预期行为？ | 用户描述、错误消息 |
| When | 何时开始？持续还是间歇？ | 日志、监控 |
| Where | 哪个环境/服务/模块/行？ | 堆栈跟踪 |
| Who | 影响哪些用户？范围？ | 用户报告、访问日志 |
| Why | 最近有什么变更？ | `git log`、部署历史 |
| How | 精确复现步骤？ | 用户步骤、自动化测试 |

关键维度（What/Where/How）缺失时，**停下来询问**。

#### 故障分类

| 类型 | 示例 | 主要策略 |
|------|------|---------|
| 崩溃 | 服务挂起、进程退出、OOM | 日志、core dump、资源使用 |
| 功能 | 逻辑错误、功能不匹配 | 代码追踪、输入输出对比 |
| 性能 | 响应慢、超时 | Profiling、瓶颈分析 |
| 间歇 | 偶发、难复现 | 增加日志、扩大观察窗口 |
| 数据 | 不一致、丢失、损坏 | 数据库查询、数据流追踪 |
| 配置 | 环境差异、参数错误 | 配置对比、环境对比 |

#### 5-Why 根因分析

找到直接原因后，反复问"为什么"直到可操作的根因：

```
接口返回 500
  -> 为什么？数据库连接超时
  -> 为什么？连接池耗尽
  -> 为什么？慢查询占用连接
  -> 为什么？大表缺少索引
  -> 根因：缺少随数据增长的索引管理机制
```

### 阶段 2：模式分析（只读）

1. 找到同代码库中类似的正常工作代码
2. 完整阅读参考实现
3. 列出所有差异
4. 理解依赖关系

### 阶段 3：假设、验证和置信度评估（只读）

#### 置信度评估

| 证据类型 | 权重 |
|---------|------|
| 日志精确匹配 | 30% |
| API 响应数据 | 30% |
| 成功复现 | 35% |
| 确定代码缺陷 | 40% |
| 确定性调用链 | 35% |
| 数据库查询结果 | 25% |
| 静态代码分析 | 15% |
| Git blame 关联 | 15% |
| 经验判断 | 10% |

#### 决策阈值

| 置信度 | 行动 |
|--------|------|
| >= 80% 高 | 进入修复阶段 |
| 50-79% 中 | 建议补充数据，或获用户明确批准 |
| < 50% 低 | 继续调查，**不提修复方案** |

### 阶段 4：实施修复

**门控：** 置信度 >= 高，或用户明确批准。

1. **修复前**：读编码规范、读测试规范、确认范围
2. **修复**：创建失败测试（走 TDD）、单一修复、验证
3. **修复后**：规则自检、回归测试、风险评估、追加修复记录
4. **修复失败**：< 3 次回到阶段 1；**>= 3 次停下来——可能是架构问题，与用户讨论**

## 排查报告模板

修复前必须生成结构化报告并保存到 `docs/agent-rules/reports/YYYY-MM-DD-<issue>-troubleshooting.md`，包含：问题摘要、5W1H、调查过程、根因分析（5-Why）、关键证据（必须原文引用）、置信度评估、修复建议、预防建议。

## 停止并求助协议

以下情况**立即停止操作并询问用户**：
- 需要真实数据但没有
- 置信度 < 50% 且无更多调查路径
- 第 3 次修复失败
- 即将修改根因范围外的代码
- 间歇性故障无法复现

## 红旗（立即停止）

- "先快速修复，以后再调查"
- "试着改改 X 看看行不行"
- 没追踪数据流就提方案
- "再试一次"（已试 2 次以上）
- 转述日志而非原文引用
- 跳过 5W1H 直接看代码
