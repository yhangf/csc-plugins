# 变更影响矩阵

不同触发事件波及不同文件。不要只看增量事实，要看事实会影响哪些知识层。

## 触发事件 → 波及文件

| 触发事件 | active-*.md | memory/feedback | memory/project | 说明 |
|---------|:--:|:--:|:--:|------|
| 用户纠正 AI 错误 | ● | ● | ○ | feedback 记录纠正详情和正确做法 |
| 用户澄清非显而易见做法 | ● | ● | ○ | 这种澄清是宝贵的领域知识 |
| 用户说明隐含约束 | ● | ● | ○ | "这个接口 QPS 不能超过 100" |
| 发现代码库关键模式 | ● | ○ | ● | project 记录模块结构、数据流 |
| 发现模块运作方式 | ● | ○ | ● | "认证流程在这个文件，逻辑是..." |
| 做出架构/实现决策 | ● | ○ | ● | 记录决策 + 为什么选这个方案 |
| 完成任务步骤 | ● | ○ | ○ | 仅更新进度 |
| 遇到阻塞/等待用户输入 | ● | ○ | ○ | 记录阻塞点和需要的输入 |
| 发现 memory 间矛盾 | ○ | ● | ● | 修正矛盾，合并为一致结论 |
| 环境变量/配置变更 | ● | ○ | ● | project 记录配置含义和默认值 |

> ●=必须写 ○=视情况（仍需判断，有价值的才写）

## "三处都补"原则（最易漏的同步场景）

当用户纠正 AI 一个做法时，不是只记一个地方就结束了：

1. **active-*.md** → `Recent Context` 段追加纠正摘要
2. **memory/feedback** → 新建或更新详细记录（为什么错、正确做法、触发场景）
3. **已有 memory 冲突检查** → 搜索已有 memory 文件，看有没有与这次纠正矛盾的老条目 → 若有，同步修正

## 跨任务影响检查

最容易漏改的场景：一个任务的发现改变了另一个任务对同一模块的理解。

- **模块理解更新**：任务 A 中发现模块 X 的真实入口 → 检查 memory/project/ 中模块 X 的已有记录
- **方案被推翻**：任务 B 中证明任务 A 选的方案不可行 → 更新 memory/project/ 中对应决策条目
- **公共约定变更**：环境变量、命名规范、部署方式变化 → 检查所有提及该约定的 memory 文件

判断方法：这次学到的事实，是否会改变已有 memory 中某个结论？会 → 同步修正。

## 不需要写入 memory 的情况

| 情况 | 原因 |
|------|------|
| AI 可在 1-2 轮内独立解决的问题 | 无需人为介入的知识不稀缺 |
| 纯业务逻辑变更（CRUD、UI 调整） | 可从代码推导 |
| 一次性的临时操作（脚本、数据修复） | 放 active-*.md 即可 |
| 显而易见的错误（typo、缺 import） | 无复用价值 |
| 标准设计模式、通用最佳实践 | 每个 AI 都默认知道 |

## 参考标准

| 规则 | 说明 |
|------|------|
| 合并优于追加 | 旧事实被推翻时改旧条目 |
| 删除优于保留 | 过期、已解决的条目删掉 |
| 绝对时间 | 永远不写 "今天""最近" |
| 一条一事 | 一个条目只讲一个事实 |
| 受众不混 | active-*.md、feedback、project 各司其职 |

详细编辑规则见 `editing-rules.md`。
