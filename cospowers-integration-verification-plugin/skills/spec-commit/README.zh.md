# Spec-Commit — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

提供**一站式 AI-Native Git 提交能力**，统一提交规范和执行流程。用户请求提交时，AI 自动完成知识合规分析、变更分析、规范检查、分支管理、提交操作、推送处理和可选的 MR 创建。

## 激活时机

用户说"帮我提交代码"、"push 代码"、"创建分支"、"创建 MR"等涉及 git commit / push / merge request 的请求时自动激活。

## Part 1：规范基线

### 保护分支

**禁止直接提交和推送**的分支：`main`、`master`、`develop`、`release/*`、`m-feature-*`

在保护分支上**必须**先创建新分支再提交。

### 分支命名规范

| 当前分支 | 新分支命名格式 |
|---------|--------------|
| m-feature-xxx | feature-xxx-自定义名 |
| develop | develop-自定义名 |
| master | master-自定义名 |
| release/* | release-自定义名 |
| main | main-自定义名 |

- "自定义名"根据变更内容自动生成（kebab-case）
- **总长度不超过 50 字符**，超过必须缩写（保留核心动词+关键名词、使用缩写、最多 2-3 个 kebab 段）

### 提交类型

| 类型 | 含义 |
|------|------|
| `[ADD]` | 新功能 |
| `[OPT]` | 优化/改进 |
| `[UPDATE]` | 调整/适配（依赖升级等） |
| `[TAG]` | 目录结构调整、注释更新 |
| `[FIX]` | Bug 修复 |
| `[REFACTOR]` | 代码重构 |
| `[DOCS]` | 文档变更 |

### AI 标签规则

- **默认标签**：`[AI-COMMIT]`
- **角色标签优先**：角色指定的标签（如 `[AI-SPEC-FIRST]`）覆盖默认
- **互斥**：一条 commit 只能有一个 AI 标签
- **格式**：`[AI-Tag][Commit-Type] 描述`

### Commit Message 完整模板

```
[AI-Tag][Type] 简要描述

##### 问题描述
##### 改动思路
##### 备选方案
##### 数据模型影响
##### REST API影响
##### 安全影响
##### 消息通知影响
##### 性能影响
##### 部署影响
##### 文档影响
##### 做了哪些测试
##### BUG ID
##### 知识合规报告
##### Checklist
Co-Authored-By: Claude Code <noreply@anthropic.com>
```

- **必填**：问题描述、改动思路、测试、Checklist
- **按需填写**：备选方案、数据模型影响、REST API 影响、BUG ID
- **无影响写 NA**：安全/消息/性能/部署/文档影响
- **`[FIX]` 类型**：问题描述需更详细（复现条件、影响范围），BUG ID 必填
- **知识合规报告**：由 Step 0 自动填写，包含涉及文档数、合规结果、规则统计和文档明细；如项目未配置知识库则填 NA

## Part 2：执行流程

### Step 0：知识合规分析（SCC/SCP/SKYOPS 项目）

> 非上述项目跳过此步骤。

提交前自动从 Daedalus 知识中枢拉取所有适用的知识文档，逐条检查本次代码变更是否合规。

#### 0.1-0.5 准备阶段

1. **获取变更**：`git diff HEAD`
2. **检测仓库主语言**：按文件数量统计（Python > Go > JS/TS），驱动后续文档拉取范围
3. **Python 版本检测**（Python 项目）：通过 `.python-version`、`pyproject.toml`、shebang、语法特征综合判定 Py2/Py3
4. **拉取显式知识文档**：按 scope（`通用`、`python`、`python3`、`python2`、`go`）并行拉取所有已发布的显式规范文档
5. **拉取隐式知识文档**：从 diff 中提取技术标签（MySQL/Redis/API/日志/安全等），按标签搜索隐式经验文档，去重并过滤低质量文档
6. **合并文档列表**：显式+隐式文档合并去重

#### 0.6 Phase 1：并行只读检查

每篇文档独立派生 `kb-compliance-checker` 子 agent（`phase=check-only`），逐条规则检查 diff，输出合规报告。此阶段不修改任何文件。

#### 0.7 Gate 决策

| 结果 | 动作 |
|------|------|
| 全部 PASS | 合规通过，继续提交流程 |
| 存在 BLOCKED | **停止**，输出激进违规清单，询问用户是否继续 |
| 存在 can-fix（无 blocked） | 进入 Phase 2 自动修复 |

#### 0.8 Phase 2：串行修复

对需要修复的文档，**逐个串行**启动 `kb-compliance-checker`（`phase=check-and-fix`），只修复保守类违规（命名、格式、类型注解、import 排序等），激进类违规（错误处理、日志、逻辑、安全等）标记为 blocked 阻塞。

#### 0.9 最终验证

修复完成后重新并行 Phase 1，确认无残留违规。

#### 0.10 输出合规报告

合规通过后，在 commit message 中自动填写 `##### 知识合规报告` 章节，包含文档数量、规则统计、通过/修复/阻断明细。

### Step 1-8：提交执行流程

| 步骤 | 内容 |
|------|------|
| 1 | 收集上下文（git status + diff + branch + log） |
| 2 | 检测保护分支 |
| 3 | 保护分支则创建新分支（含长度校验） |
| 4 | 分析变更 + 逐个 git add（排除敏感文件，不用 `git add -A`） |
| 5 | 组装 commit message（上库规范模板，含知识合规报告） |
| 6 | 执行 git commit（HEREDOC 格式） |
| 7 | 推送处理（禁止自动推送，需用户确认，禁止推到保护分支） |
| 8 | 可选：创建 MR（从 git remote 提取域名、读 token、确定目标分支、准备描述、调用 GitLab API） |

## 保守修复 vs 激进修复

| 保守修复（自动执行） | 激进修复（标记 blocked） |
|---------------------|------------------------|
| 命名规范、格式化、类型注解 | 错误处理、日志 |
| Import 排序、缺失 docstring | 逻辑变更、API 签名变更 |
| 魔法值提取为常量 | 架构约束、安全 |

> 判断原则：修改后可能改变运行时行为的 → 激进修复，绝不自动执行。

## 禁止操作清单

- 提交到保护分支
- 直接推送保护分支
- 使用 `git add -A` 或 `git add .`
- 提交 `agent-rules/` 目录下的文件
- 未经用户确认自动推送
- 不同目的的变更混入单次提交
- commit message 缺 AI 标签或 Co-Authored-By
- 使用 `git push --force`
- 硬编码 GitLab 域名
- 提交敏感文件（.env、credentials 等）
- 跳过分支名长度校验
- 未经用户请求使用 `git commit --amend`

## 角色扩展

特定角色可在此基线上扩展：自定义 AI 标签、附加提交块、额外检查项。角色扩展与本 skill 冲突时，**角色扩展优先**。
