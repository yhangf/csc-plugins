# 单元测试规范（通用）

## 用途

本文档供 AI 执行两类任务：**代码生成**（写测试前对照规则）、**代码 Review**（逐条扫描输出违规）。

违规报告格式：`[规则ID] <文件>:<行号> — <描述>`

强制等级：`[强制]` 必须遵守 / `[建议]` 推荐遵守

---

## 测试策略决策树

```
函数属于哪类？
  业务强相关（对外接口、provider 方法、公共基础库）→ 3-High，必须写单测
  中等复杂（30-80 行 或 ≥2 个入参）              → 2-Medium，建议写单测
  简单函数（<30 行 且 ≤1 个入参，业务弱相关）    → 1-Low，酌情

Python API 服务写哪一层？
  正常场景 → Controller 接口（端到端验证）
  异常场景 → Provider 方法（边界与错误验证）
  内部方法 → 不单独写，由上层接口覆盖
```

---

## 规则详情

### 优先级与覆盖

**UT-PRIO-01** `[强制]` 以下函数必须写单测（3-High），且必须覆盖所有异常逻辑：
- 业务强相关函数：后台服务对外接口、Phoenix 框架 provider 对外接口
- 基础库函数：本服务公共方法、sf-libs、sf-go-libs 等

违规信号：3-High 函数无单测文件；单测只覆盖正常场景、异常分支无任何用例。

---

**UT-PRIO-02** `[强制]` 追求场景覆盖率（特别是异常场景），不以代码覆盖率为唯一目标；每个等价类至少有一条用例，异常场景覆盖完整后覆盖率自然提升。

违规信号：单测函数只有一个正常用例；所有 case 都是 happy path；异常分支（参数非法、外部调用失败、回滚逻辑）无测试。

---

### 四步骤规范

**UT-STEP-01** `[强制]` 每个测试按以下四步编写，缺一不可：

```
1. 注释   — 说明被测函数名、等级、测试思路（场景类型 + 期望结果）
2. 构造参数 — 按业务场景构造有意义的数据，不能用随机或固定"永远正确"的值
3. 打桩   — Mock 外部依赖，只 Mock 当前用例需要的
4. 断言   — 验证返回值 + 验证 Mock 对象（调用次数、调用参数）
```

违规信号：无注释说明测试思路；构造数据全是占位符；无任何断言（只跑覆盖率）；有 Mock 但从未校验 `call_count`/`assert_called_with`。

---

### 注释规范

**UT-CMT-01** `[强制]` 每个测试函数必须有结构化注释，包含：被测函数名、被测函数等级、测试思路（场景类型 + 期望结果）。

```python
# ✅ Python 注释模板
def test_create_alarm_strategy_raises_error_when_param_invalid(
        self, mock_create, mock_list):
    """
    告警策略参数校验失败
    被测函数名: create_alarm_strategy
    被测函数等级: 3-High
    单测函数思路（异常场景）:
    （临界值）通知人个数超出限制 期望：返回 操作失败，参数输入无效
    （临界值）converge 值超出临界值 期望：返回 操作失败，参数输入无效
    （异常值）dims 非法，告警维度是业务但对象长度不为 1 期望：返回 操作失败，参数输入无效
    """
```

```go
// ✅ Go 注释模板
/*
* 被测函数名: render
* 被测函数等级: 3-High
* 单测函数功能描述: 利用通知模板生成告警通知信息
* 单测函数测试思路:
* 1、（微信通知模板）指标告警触发 期望：返回 XXXXX（XXXX）XXX 告警\n...
* 2、（微信通知模板）指标告警升级 期望：返回 XXXXX（XXXX）升为 XXX 告警\n...
*/
func TestRender_WhenMetricAlarmTriggered(t *testing.T) { ... }
```

---

**UT-CMT-02** `[强制]` 测试函数命名遵循 `test_<主体>_<行为>_when_<条件>`（Python）/ `Test<主体>_<行为>_When<条件>`（Go）格式。

违规信号：`TestCreate`、`test_user`、`test_case1` 等无法判断测试场景的命名。

```python
❌ def test_create(self): ...
❌ def test_case2(self): ...

✅ def test_create_trust_raises_error_when_trustor_mismatch(self): ...
✅ def test_list_trusts_returns_expired_status_when_deadline_passed(self): ...
```

```go
❌ func TestCreate(t *testing.T) {}

✅ func TestDeleteVM_ReturnsError_WhenVMNotFound(t *testing.T) {}
✅ func TestCreateReport_Succeeds_WhenAllParamsValid(t *testing.T) {}
```

### 测试数据

**UT-DATA-01** `[强制]` 测试数据必须按业务场景构造，不能使用随意的固定值或永远不会触发边界的数据；时间相关场景必须基于当前时间动态计算。

违规信号：`expires_at = "2020-01-01"`（永远过期，无法验证时间边界）；`id = "123"`（无业务意义）。

```python
# ❌ 静态时间，永远都是"过期"，无法验证临界边界
trust["expires_at"] = "2020-01-01 00:00:00"

# ✅ 基于当前时间动态计算，验证"刚过期"边界
local_now = datetime.datetime.now()
local_now_1min_before = local_now - relativedelta(minutes=1)
trust["expires_at"] = local_now_1min_before.strftime(TIME_FORMAT)
```

---

**UT-DATA-02** `[建议]` 测试数据贴近真实业务格式（真实格式的 UUID、时间戳、业务名称）；公共测试数据抽取为工厂函数，用例中只修改差异字段。

```python
# ✅ 复用基础数据，只修改当前用例需要的字段
create_trust_param = copy.deepcopy(trust_data.create_trust)   # 基础模板
create_trust_param['user_id'] = data_utils.rand_uuid_hex()    # 只改需要的字段
```

### 断言规范

**UT-ASSERT-01** `[强制]` 每个用例必须有完整断言：返回值断言 + Mock 对象校验（调用次数、调用参数）；禁止无断言的"空跑"用例。

违规信号：测试函数无任何 `assert`/`So`/`assertEqual`；有 Mock 但无 `assert_called_with`/`assert_not_called`。

```python
# ❌ 无断言，只跑覆盖率
def test_create_trust(self, ...):
    self.trust_api.create_trusts(self.req, params)

# ✅ 验证返回值 + 验证 Mock 调用参数
def test_create_trust(self, mock_run_async, ...):
    result = self.trust_api.create_trusts(self.req, params)
    self.assertEqual(result, trust_data.update_trusts_return)
    self.assertEqual(mock_run_async.call_count, 4)          # 验证调用次数

# ✅ 验证 Mock 未被调用
mock_update_user_trusts.assert_not_called()

# ✅ 验证 Mock 调用参数精确匹配
mock_delete_region_trusts.assert_called_with(
    self.req, [trust_data.trust_region], common.DEFAULT_USER_ID,
    [common.DEFAULT_MSP_NAME, common.DEFAULT_GLOBAL_ADMIN_NAME])
```

---

**UT-ASSERT-02** `[强制]` 验证异常场景时必须用精确的异常类型断言，不能用宽泛的 `Exception`；异常消息/返回码等关键信息也需断言。

```python
❌ self.assertRaises(Exception, self.trust_api.create_trusts, ...)

✅ self.assertRaises(exception.VisibleExNoPermissionTrust,
                   self.trust_api.create_trusts,
                   self.req, create_trust_param)
```

---

### Mock 规范

**UT-MOCK-01** `[强制]` 不滥用 Mock：只 Mock 当前用例真正需要的外部依赖；每个 Mock 对象必须进行调用断言（调用次数、调用参数或 `assert_not_called`）。

违规信号：测试函数 `@mock.patch` 了 5 个函数但只用到 2 个；Mock 函数从未做调用断言。

---

**UT-MOCK-02** `[强制]` 测试用例前后必须有前置准备和后置清理，用例间无依赖、不破坏测试环境；Python 使用 `setUp`/`tearDown`，Go 使用 `defer patches.Reset()` 或 `t.Cleanup`。

违规信号：前一个用例的 Mock side_effect 影响下一个用例；Go 测试忘记 `patches.Reset()`。

```python
# ✅ 每个 case 重置 side_effect，防止污染
mock_update_user_trusts.side_effect = None   # 重置前一个 case 的副作用
mock_update_user_trusts.return_value = trust_data.update_trusts_return
```

---

### 用例设计

**UT-DESIGN-01** `[建议]` 用例设计遵循以下方法，优先覆盖异常场景：

| 设计方法 | 说明 | 示例 |
|---------|------|------|
| 边界值 | 临界值-1、最小临界值、中间值、最大临界值、临界值+1 | 通知人数上限前后 |
| 等价类 | 有效等价类 + 无效等价类 | 合法参数 vs 非法参数 |
| 容量类 | 空载、部分数据、满载 | 空列表、单条、满配额 |
| 异常场景 | 错误类型、参数缺失、外部调用失败 | 数据库超时、参数为空 |
| 用户场景 | 真实业务流程端到端 | 租户申请代维完整流程 |

---

### 流水线要求

**UT-CI-01** `[强制]` 单测通过率 100% 作为转测标准之一；每次上库自动执行单元测试；失败用例只因 bug 失败，不存在"环境不同结果不同"的不稳定用例。

违规信号：单测在 CI 上偶发失败；测试依赖本地环境变量/外部网络；跨用例存在状态依赖。

## Code Review Checklist

逐项扫描，违规输出 `[规则ID] <文件>:<行号> — <描述>`。

**优先级与覆盖**
- [ ] UT-PRIO-01 — 3-High 函数是否有单测；异常分支是否覆盖
- [ ] UT-PRIO-02 — 是否只有正常场景用例，缺失异常场景

**四步骤**
- [ ] UT-STEP-01 — 是否有注释/构造参数/打桩/断言四步，缺少断言是高压线

**注释**
- [ ] UT-CMT-01 — 测试函数是否有被测函数名、等级、测试思路（场景+期望）
- [ ] UT-CMT-02 — 测试函数命名是否符合 `test_主体_行为_when_条件` 格式

**测试数据**
- [ ] UT-DATA-01 — 时间相关数据是否动态计算；数据是否有业务意义（非随意填写）
- [ ] UT-DATA-02 — 重复数据是否抽为工厂函数（建议）

**断言**
- [ ] UT-ASSERT-01 — 是否有返回值断言；有 Mock 是否校验调用次数/调用参数
- [ ] UT-ASSERT-02 — 异常场景是否用精确异常类型断言，而非宽泛 Exception

**Mock**
- [ ] UT-MOCK-01 — 是否有无用的 Mock（@mock.patch 但从未使用/断言）
- [ ] UT-MOCK-02 — 用例间是否有状态污染；Mock side_effect 是否在每个 case 前重置

**用例设计**
- [ ] UT-DESIGN-01 — 是否覆盖边界值/等价类/异常场景（建议）

**流水线**
- [ ] UT-CI-01 — 单测是否在不同环境下结果稳定；是否依赖外部网络/本地文件

## 附录：典型单测结构示例

### Python Controller 正常场景
```python
@mock.patch("{driver}.create_region_trusts".format(driver=TRUST_DRIVER_PATH))
@mock.patch("{driver}.update_user_trusts".format(driver=TRUST_DRIVER_PATH))
def test_create_trust_succeeds_when_all_params_valid(
        self, mock_update, mock_create_region):
    """
    创建代维授权成功
    被测函数名: create_trusts
    被测函数等级: 3-High
    单测函数思路（正常场景）:
    租户申请代维授权（msp 和 global admin），2 个数据中心，
    创建授权成功，并成功发送通知
    """
    # 1. 构造参数
    params = copy.deepcopy(trust_data.create_trust)

    # 2. 打桩
    mock_create_region.return_value = trust_data.region_mgmt_success_return
    mock_update.return_value = trust_data.update_trusts_return

    # 3. 执行
    result = self.trust_api.create_trusts(self.req, params)

    # 4. 断言返回值 + 断言 Mock 调用
    self.assertEqual(result, trust_data.update_trusts_return)
    mock_update.assert_called_once()
```

### Python Provider 异常场景
```python
def test_create_trust_raises_error_when_trustor_mismatch(self, ...):
    """
    被测函数名: create_trusts  等级: 3-High
    （异常值-参数异常）代维租户和请求租户不一致 期望：返回 VisibleExNoPermissionTrust
    """
    # 构造 user_id 不一致
    params = copy.deepcopy(trust_data.create_trust)
    params['user_id'] = data_utils.rand_uuid_hex()

    # 断言抛出精确异常
    self.assertRaises(exception.VisibleExNoPermissionTrust,
                      self.trust_api.create_trusts,
                      self.req, params)
```


通用补充：
## 测试优先级
- **[强制]** 将函数按 `3-High`/`2-Medium`/`1-Low` 进行优先级划分
- **[强制]** 对 `3-High` 级函数必须编写单元测试并覆盖所有异常逻辑
- **[建议]** 对 `2-Medium` 级函数建议编写单元测试
- **[建议]** 对 `1-Low` 级函数按需编写单元测试
- **[建议]** 按函数行数和入参数判定优先级，行数>80或入参>=3倾向 `3-High`
- **[建议]** 将行数 30-80 或入参>=2 的函数归为 `2-Medium`

## 测试场景
- **[强制]** 覆盖正常、边界和异常三类必测场景
- **[强制]** 设计等价类、边界值、容量和正交组合测试用例
- **[建议]** 按业务场景设计用例，优先覆盖业务关键路径

## 覆盖率与流水线要求
- **[强制]** 在每次上库时自动执行单元测试
- **[强制]** 将所有单测发现的问题记录到单元测试数据度量中
- **[强制]** 将单测通过率 100% 作为流水线转测标准之一
- **[建议]** 控制单测整体执行时间，尽量在 10 分钟内完成

## 禁止项
- **[强制]** 禁止测试依赖执行顺序或存在测试间依赖
- **[强制]** 禁止用例破坏测试环境，要求前置准备和后置清理
- **[强制]** 禁止出现无断言或空断言的测试用例

## 断言与 Mock 校验
- **[强制]** 验证返回值与输出性参数是否符合预期
- **[强制]** 验证必要的逻辑执行情况，使用 mock 的断言进行校验 `call_count`、`assert_call_with`
- **[强制]** 对 mock 对象的调用次数和入参进行断言检查 `call_count`、`assert_call_with`
- **[建议]** 避免滥用 mock，优先使用真实或轻量替代打桩

## 用例设计与稳定性
- **[强制]** 按步骤编写单测：注释、构造参数、打桩、断言
- **[强制]** 保证用例在不同环境下执行结果一致，不依赖开发环境
- **[强制]** 为每个用例编写清晰可读的命名和测试说明
- **[建议]** 构造测试数据贴合业务场景并考虑临界时间点
