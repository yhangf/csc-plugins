# 模块 M5：国际化 i18n 设计

## 模块概述

**审查目的**：确保设计文档在多语言支持、翻译规范化、数据持久化和跨产品同步方面遵守国际化设计标准。

**适用范围**：所有类型的设计文档（总体设计、概要设计、技术需求设计）。本模块为必选审查模块。

**说明**：本模块共 7 个检查项，分为 5 个类别。

**规范链接**：内部 i18n 规范详见 `http://docs.sangfor.org/x/F6GYDW`，内有详细的规范说明和示例。

---

## 00. 翻译文本统一标识符规范

### 0.1 翻译文本不能硬编码

**检查内容**：所有跟国际化、本地化相关的都不允许直接在代码或配置中硬编码，包含常规的文案、日期、数字、货币等信息。对于这类信息改成统一的标识符，通过语言包最终映射，在运行时解析后才显示出真实的文案。

**评判标准**：
- 通过：设计中明确说明翻译文本使用标识符 + 语言包机制，无硬编码
- 不通过：设计中出现硬编码的可翻译文本，或未提及翻译机制
- 不适用：产品明确不需要国际化支持，已说明原因

**通过示例**：
> "所有用户可见文案通过 i18n 标识符（如 `error.file_not_found`）引用，运行时从 PO 语言包解析，不直接在代码中写任何语言的文案。"

**不通过示例**：
> "错误提示信息直接在代码中返回 `'文件未找到'`。"

## 01. 翻译文本规范

### 1.1 翻译机制支持单复数

**检查内容**：在一条翻译文案中支持单复数。代码调用的翻译接口接受要翻译的数量和占位符，能够自动解析单复数格式，返回正确的词条，对接口调用者透明。

**评判标准**：
- 通过：翻译机制支持单复数自动处理，对调用者透明
- 不通过：翻译机制未考虑单复数，或需要调用者自行处理
- 不适用：产品无单复数文案场景

**通过示例**：
> 英文 PO 文件：
> ```
> msgid "found_files"
> msgid_plural "found_files"
> msgstr[0] "Found %d file"
> msgstr[1] "Found %d files"
> ```

**不通过示例**：
> 在代码中写 `if count == 1: return "1 file" else: return f"{count} files"`（手���处理单复数）

### 1.2 时间显示

**检查内容**：不同国家、地区对时间格式有不同的显示方式。如果业务中代码生成对国家、地区敏感的时间信息并对客户显示，需要设计对应的翻译接口支持该场景。

**评判标准**：
- 通过：设计中考虑了时区和地区格式化的时间显示接口
- 不通过：涉及时间显示但未考虑国际化格式
- 不适用：无时间显示需求

### 1.3 货币和数字单位显示

**检查内容**：不同国家对货币和数字单位的显示都有所区别。翻译模块对业务层提供通用、透明的接口。

**评判标准**：
- 通过：有通用的货币/数字格式化接口设计
- 不通过：涉及货币/数字显示但未考虑国际化
- 不适用：无货币/数字显示需求

## 02. 映射文件规范

### 2.1 映射文件格式

**检查内容**：只能使用 PO 格式的映射文件。

**评判标准**：
- 通过：设计中明确使用 PO 格式
- 不通过：使用非 PO 格式（如 JSON、XML 自定义格式）
- 不适用：无国际化需求

---

## 03. 数据持久化规范

### 3.1 翻译数据持久化规范

**检查内容**：当有翻译文本持久化的业务（比如操作日志保存在数据库），需要持久化多份数据，包括翻译前的信息和翻译后的文本。保存翻译前的信息是为了语言切换后可以再次被动态翻译；保存翻译后的文本可用于跨语言查询。

**评判标准**：
- 通过：同时保存翻译前标识符和翻译后文本，满足语言切换和跨语言查询需求
- 不通过：只保存翻译后文本，无法支持语言切换；或只保存标识符，无法支持跨语言查询
- 不适用：无翻译文本持久化需求

**通过示例**：
> 操作日志表设计：
> - `msg_key VARCHAR(128)` — 保存翻译标识符，如 `action.file_upload`
> - `msg_params JSON` — 保存翻译参数
> - `msg_text VARCHAR(512)` — 保存当前语言下的翻译后文本，用于跨语言查询

## 04. 翻译信息同步规范

### 4.1 跨产品翻译信息同步规范

**检查内容**：在跨产品的解决方案中，需要在设计时考虑如何同步翻译信息，让所有翻译信息都能在各个产品中进行动态翻译，对用户提供统一的语言体验。需满足：
1. 所有产品增加 API，提供同步翻译资源包的功能，让其他产品在对接时能够获取本产品的翻译资源包
2. 在业务 API 中，如果返回了需要翻译的字段，不直接返回翻译后的文本，而是将待翻译的标识符和变量等信息返回

**评判标准**：
- 通过：有翻译资源包同步 API，业务 API 返回标识符而非翻译后文本
- 不通过：跨产品方案中无翻译同步设计，或 API 直接返回翻译后文本
- 不适用：非跨产品方案，已明确说明
