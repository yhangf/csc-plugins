# 安全规范

## 用途

本文档供 AI 执行两类任务：**需求设计**（新接口/新特性设计时对照安全基线）、**代码 Review**（评估实现是否满足安全要求）。

违规报告格式：`[规则ID] <接口/模块路径> — <描述>`

强制等级：`[强制]` 必须遵守 / `[建议]` 推荐遵守

---

## 规则详情

### 界面安全

**SEC-UI-01** `[强制]` 敏感信息展示：页面仅展示必要的信息，敏感信息默认脱敏（口令、密钥、金额、身份证、电话号码、邮箱、姓名等）。

违规信号：页面直接展示明文敏感信息；敏感字段未做脱敏处理。

```
敏感数据分级分类：
  密码：掩码输入（易用性要求高的场景，可提供小眼睛查看）
  其他敏感信息（身份证、电话号码、邮箱、姓名、金额等）：
    默认掩码展示（后端返回即掩码）
    点击查看后二次请求返回明文
```

---

**SEC-UI-02** `[强制]` 默认安全：安全配置默认勾选；关闭安全配置或开启不安全配置时，必须进行安全风险提示。

违规信号：安全配置默认未勾选；关闭安全配置时无风险提示弹窗。

---

### 接口安全

**SEC-API-01** `[建议]` 接口/端口暴露面最小化，端口监听范围最小化，限制接口调用方来源 IP。

违规信号：接口未做 IP 白名单限制；端口监听 0.0.0.0 而非指定网段。

```
访问控制策略：
  谁（应用、微服务）能调这个接口？
  端口监听范围是否最小化？
  常见策略：IP 地址绑定、MAC 地址绑定等
```

---

**SEC-API-02** `[强制]` 接口有认证机制，防止未授权访问（包括内网）；新增对外开放端口和服务必须满足身份认证模块相关要求。

违规信号：内网接口无任何认证机制；新增对外开放端口未接入身份认证。

---

**SEC-API-03** `[强制]` 接口有角色权限校验，只能操作角色相关的接口，防止租户内垂直越权和租户间水平越权。

违规信号：租户 A 可访问租户 B 的数据；普通用户可执行管理员操作。

---

**SEC-API-04** `[强制]` 使用安全传输协议（HTTPS）；跨信任域传输密钥、密码等 L4 级别敏感信息时，应加密后再传输。

违规信号：接口使用 HTTP 明文传输；跨信任域传输密钥未做二次加密。

---

**SEC-API-05** `[强制]` 接口返回参数最小化，不返回多余字段；敏感信息应在后台进行脱敏处理。

违规信号：接口返回未使用的字段；返回数据中包含明文敏感信息。

---

**SEC-API-06** `[强制]` 存在会话标识的关键 cookie，必须设置 `httpOnly` 和 `secure` 参数。

违规信号：会话 cookie 未设置 `httpOnly`；会话 cookie 未设置 `secure`。

```
Set-Cookie: sessionId=xxx; HttpOnly; Secure; SameSite=Strict
```

---

### 认证授权

**SEC-AUTH-01** `[强制]` 为接口实现认证机制以防止未授权访问。

违规信号：接口无任何认证校验即可访问。

---

**SEC-AUTH-02** `[强制]` 对新增对外开放端口和服务强制满足身份认证要求。

违规信号：新增对外服务未接入统一身份认证系统。

---

**SEC-AUTH-03** `[强制]` 对接口实施基于角色的权限校验，防止租户间水平越权和租户内垂直越权。

违规信号：用户可访问或操作非授权范围的资源。

---

**SEC-AUTH-04** `[强制]` 为关键会话 cookie 设置 `httpOnly` 和 `secure` 标志。

违规信号：会话 cookie 缺少 `httpOnly` 或 `secure` 属性。

---

### 数据安全

**SEC-DATA-01** `[建议]` 数据收集遵循正当透明、授权同意原则，在允许的范围内收集（如 IP 白名单限制）。

违规信号：未获得用户授权即收集数据；收集范围超出业务必要。

---

**SEC-DATA-02** `[建议]` 遵循数据最小化原则，只采集满足业务所需的最少数据，不多收集。

违规信号：采集了业务不需要的用户字段。

---

**SEC-DATA-03** `[建议]` 数据采集有校验身份，对信息的来源进行判断，能对伪造来源的信息予以鉴别。

违规信号：数据采集无来源校验，可被伪造数据欺骗。

---

**SEC-DATA-04** `[建议]` 密码、密钥等机密信息应掩码输入。

违规信号：密码输入框明文显示。

---

**SEC-DATA-05** `[强制]` 记录数据采集操作的审计日志。

违规信号：数据采集动作无日志记录。

---

**SEC-DATA-06** `[强制]` 使用 HTTPS 安全协议传输敏感和机密数据；密钥、密码等重要数据应当加密后再传输。

违规信号：使用 HTTP 传输敏感数据；明文传输密钥或密码。

---

**SEC-DATA-07** `[建议]` 重要数据应有完整性校验，防止传输过程中被篡改（如签名）。

违规信号：重要数据传输无签名或校验机制。

---

**SEC-DATA-08** `[强制]` 浏览器页面展示数据需要对特殊字符做 HTML 转义输出，防止 XSS。

违规信号：用户输入直接渲染到页面未做转义。

```
✅ 正确：对用户输入做 HTML 转义后再输出到页面
❌ 错误：直接将用户输入拼接为 innerHTML
```

---

**SEC-DATA-09** `[强制]` 敏感数据脱敏或加密后再存储在数据库或配置文件中。

违规信号：数据库或配置文件中存在明文密码、密钥、身份证等敏感信息。

---

**SEC-DATA-10** `[强制]` 数据存储组件有身份访问控制。

违规信号：数据库、对象存储等组件无访问控制，可被任意访问。

---

**SEC-DATA-11** `[强制]` 密钥、重要配置文件、虚拟机镜像等重要数据存储时有完整性校验。

违规信号：重要文件无完整性校验（如无 hash 校验）。

---

**SEC-DATA-12** `[建议]` 检测到完整性受到破坏时，采取必要的恢复措施。

违规信号：完整性校验失败时仅告警，无恢复机制。

---

**SEC-DATA-13** `[强制]` 记录直接对存储数据的操作行为日志。

违规信号：直接操作数据库/文件存储的行为无日志。

---

**SEC-DATA-14** `[建议]` 仅在云服务客户授权下，云服务商或第三方才具有云服务客户数据的管理权限。

违规信号：未获得客户授权即访问或管理客户数据。

---

**SEC-DATA-15** `[强制]` 访问数据有身份校验，能防止未授权访问（租户间不能越权查看）。

违规信号：租户 A 可查看租户 B 的数据。

---

**SEC-DATA-16** `[强制]` 访问数据有角色权限校验，能有效拒绝非法访问（租户内垂直越权）。

违规信号：普通角色可执行管理员操作。

---

**SEC-DATA-17** `[建议]` 校验数据权限，用户只能操作权限范围内的数据；限制敏感数据查询的频率和数据量，防止数据批量泄露。

违规信号：用户可一次查询全量敏感数据；无频率限制可被批量爬取。

---

**SEC-DATA-18** `[建议]` 重要数据在使用前先校验完整性。

违规信号：使用重要配置/密钥前未校验其完整性。

---

**SEC-DATA-19** `[建议]` 防 DOS 攻击、防命令注入等可用性防护。

违规信号：接口无限流措施；系统命令拼接用户输入。

---

### 密钥管理

**SEC-KEY-01** `[强制]` 生成密钥时保证长度和复杂度，不小于 32 位。

违规信号：密钥长度不足 32 位；使用简单字符串作为密钥。

---

**SEC-KEY-02** `[建议]` 使用推荐的密码算法生成高强度的密钥，如 AES256、RSA2048 以上。

违规信号：使用 DES、MD5 等已淘汰算法生成密钥。

---

**SEC-KEY-03** `[强制]` 密钥文件权限 400，密钥文件目录权限 600；密钥文件有完整性保护。

违规信号：密钥文件权限大于 400；密钥目录权限大于 600。

```
chmod 400 /path/to/key
chmod 600 /path/to/key/dir
```

---

**SEC-KEY-04** `[建议]` 密钥应按二层/三层密钥保护机制进行密钥保护。

违规信号：密钥明文存储，无分层保护机制。

---

**SEC-KEY-05** `[强制]` 不要将密钥回显到前端界面；首次配置时可允许查看（默认掩码），配置成功后不允许提供查看功能。

违规信号：配置完成后仍可查看密钥明文；密钥明文回显到前端。

---

**SEC-KEY-06** `[强制]` 密钥的用途唯一。

违规信号：同一密钥用于多个不同业务场景。

---

### 日志审计

**SEC-LOG-01** `[强制]` 增删改查接口有日志记录，敏感数据查询有日志记录。

违规信号：增删改操作无日志；敏感数据被查询但无审计记录。

---

**SEC-LOG-02** `[强制]` 审计覆盖到每个用户，对重要的用户行为进行审计。

违规信号：部分用户操作未纳入审计范围。

---

**SEC-LOG-03** `[强制]` 记录必要的日志要素，包括：

违规信号：日志缺少以下任一要素。

```
日志要素：
  事件发生的时间
  用户 ID
  访问发起端地址或标识（关联终端、端口、网络地址或通信设备等）
  事件类型
  被访问的资源名称
  事件的结果
```

---

**SEC-LOG-04** `[强制]` 记录所有对敏感数据的操作日志（增删查改）。

违规信号：敏感数据的操作无日志记录。

---

### 通用功能安全

**SEC-FUNC-01** `[强制]` 验证码安全：一般为 6 位，安全随机（不可预测），一次性；有效期限制（如 5 分钟内有效）。

违规信号：验证码可预测；验证码可重复使用；验证码长期有效。

```
✅ 6 位随机数字，5 分钟有效，一次性使用
❌ 4 位固定验证码，无过期时间，可重复使用
```

---

**SEC-FUNC-02** `[强制]` 文件上传：白名单限定文件类型；限制文件上传路径，不返回绝对路径；限制文件大小。

违规信号：可上传任意文件类型；返回文件绝对路径；无文件大小限制。

---

**SEC-FUNC-03** `[建议]` 文件下载：限制文件下载目录；限制文件下载频率（防 DOS）；提供可执行文件下载时，页面提供对应的 hash 值。

违规信号：可目录遍历下载任意文件；下载无频率限制；可执行文件下载无 hash。

---

**SEC-FUNC-04** `[强制]` 权限管理模块：系统应设计统一的权限控制模型，访问控制策略满足基本原则：隐式拒绝、知其所需、最小特权、职责和责任分离。

违规信号：权限模型默认允许；用户权限超出职责范围。

---

**SEC-FUNC-05** `[强制]` 数据库访问：访问数据库的 SQL 语句需要使用预编译防止 SQL 注入。

违规信号：拼接用户输入构造 SQL 语句。

```
✅ PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?")
❌ Statement stmt = conn.createStatement("SELECT * FROM users WHERE id = " + userId)
```

---

**SEC-FUNC-06** `[强制]` XML 解析：应用程序解析 XML 文档时，禁止加载外部实���（防范 XXE 攻击）。

违规信号：XML 解析器未禁用外部实体加载。

```
✅ factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true)
❌ 使用默认 XML 解析器配置，允许 DOCTYPE 声明
```

---

**SEC-FUNC-07** `[强制]` 服务器代理客户请求：禁止除 HTTPS 和 HTTP 之外的协议；根据请求需要设置请求白名单，拒绝白名单之外的代理请求。

违规信号：代理支持 `file://`、`gopher://` 等协议；无白名单限制可代理任意 URL。

---

**SEC-FUNC-08** `[强制]` 系统命令执行：执行系统命令的参数来源于客户输入时，需要对参数做好过滤，对敏感字符进行转义。

违规信号：用户输入直接拼接到系统命令中。

```
✅ 白名单校验参数 + 参数转义
❌ os.system("ping " + user_input)
```

---

**SEC-FUNC-09** `[建议]` 获取用户 IP：尽可能获取靠近用户端的真实 IP，可通过 Remote Address 获取用户与服务器建立 TCP 连接时的 IP，不可直接通过 HTTP 请求头 `X-Forwarded-For` 等获取可被伪造的 IP。

违规信号：直接使用 `X-Forwarded-For` 作为用户真实 IP。

```
✅ request.getRemoteAddr()（TCP 连接 IP）
❌ request.getHeader("X-Forwarded-For")（可被伪造）
```

## Code Review Checklist

逐项扫描，违规输出 `[规则ID] <接口/模块路径> — <描述>`。

**界面安全**
- [ ] SEC-UI-01 — 敏感信息是否默认脱敏展示（口令、密钥、金额、身份证、电话、邮箱、姓名）
- [ ] SEC-UI-02 — 安全配置是否默认勾选；关闭不安全配置时是否有风险提示

**接口安全**
- [ ] SEC-API-01 — 接口/端口暴露面是否最小化，是否限制调用方来源 IP（建议）
- [ ] SEC-API-02 — 接口是否有认证机制，新增对外端口是否满足身份认证要求
- [ ] SEC-API-03 — 接口是否有角色权限校验，防止水平越权和垂直越权
- [ ] SEC-API-04 — 是否使用安全传输协议；跨信任域敏感数据传输是否加密
- [ ] SEC-API-05 — 接口返回参数是否最小化，敏感字段是否脱敏
- [ ] SEC-API-06 — 关键会话 cookie 是否设置 httpOnly 和 secure

**认证授权**
- [ ] SEC-AUTH-01 — 接口是否实现认证机制
- [ ] SEC-AUTH-02 — 新增对外开放端口/服务是否满足身份认证要求
- [ ] SEC-AUTH-03 — 是否实施基于角色的权限校验
- [ ] SEC-AUTH-04 — 关键会话 cookie 是否设置 httpOnly 和 secure

**数据安全**
- [ ] SEC-DATA-01 — 数据收集前是否获得授权（建议）
- [ ] SEC-DATA-02 — 是否遵循数据最小化原则（建议）
- [ ] SEC-DATA-03 — 数据采集是否有身份校验（建议）
- [ ] SEC-DATA-04 — 密码/密钥是否掩码输入（建议）
- [ ] SEC-DATA-05 — 数据采集操作是否有审计日志
- [ ] SEC-DATA-06 — 敏感数据是否使用 HTTPS 传输
- [ ] SEC-DATA-07 — 重要数据传输是否有完整性校验（建议）
- [ ] SEC-DATA-08 — 浏览器输出前是否对特殊字符做 HTML 转义
- [ ] SEC-DATA-09 — 敏感数据存储前是否脱敏或加密
- [ ] SEC-DATA-10 — 数据存储组件是否有身份访问控制
- [ ] SEC-DATA-11 — 密钥/重要配置/镜像是否有完整性校验
- [ ] SEC-DATA-12 — 完整性校验失败时是否有恢复措施（建议）
- [ ] SEC-DATA-13 — 直接操作存储数据的行为是否有日志
- [ ] SEC-DATA-14 — 客户数据管理是否获得客户授权（建议）
- [ ] SEC-DATA-15 — 数据访问是否有身份校验防止租户越权
- [ ] SEC-DATA-16 — 数据访问是否有角色权限校验防止垂直越权
- [ ] SEC-DATA-17 — 是否限制敏感数据查询频率和数量（建议）
- [ ] SEC-DATA-18 — 重要数据使用前是否校验完整性（建议）
- [ ] SEC-DATA-19 — 是否有防 DOS 和命令注入措施（建议）

**密钥管理**
- [ ] SEC-KEY-01 — 密钥长度是否 ≥32 位
- [ ] SEC-KEY-02 — 是否使用推荐的密码算法（AES256/RSA2048）（建议）
- [ ] SEC-KEY-03 — 密钥文件权限是否为 400，目录权限是否为 600
- [ ] SEC-KEY-04 — 密钥是否按二/三层保护机制存储（建议）
- [ ] SEC-KEY-05 — 密钥是否禁止回显到前端；配置成功后是否禁止查看
- [ ] SEC-KEY-06 — 密钥用途是否唯一

**日志审计**
- [ ] SEC-LOG-01 — 增删改查接口及敏感数据查询是否有日志
- [ ] SEC-LOG-02 — 审计是否覆盖到每个用户的重要行为
- [ ] SEC-LOG-03 — 日志是否包含必要要素（时间、用户ID、地址、事件类型、资源、结果）
- [ ] SEC-LOG-04 — 敏感数据的增删查改操作是否有日志

**通用功能安全**
- [ ] SEC-FUNC-01 — 验证码是否满足随机性、一次性、有效期要求
- [ ] SEC-FUNC-02 — 文件上传是否有类型白名单、路径限制、大小限制
- [ ] SEC-FUNC-03 — 文件下载是否有目录限制、频率限制、hash 校验（建议）
- [ ] SEC-FUNC-04 — 权限控制模型是否满足最小特权、隐式拒绝原则
- [ ] SEC-FUNC-05 — SQL 语句是否使用预编译防止注入
- [ ] SEC-FUNC-06 — XML 解析是否禁用外部实体
- [ ] SEC-FUNC-07 — 代理请求是否限制协议（仅 HTTP/HTTPS）和请求白名单
- [ ] SEC-FUNC-08 — 执行系统命令时用户输入参数是否经过过滤转义
- [ ] SEC-FUNC-09 — 获取用户 IP 是否使用 Remote Address 而非请求头（建议）
