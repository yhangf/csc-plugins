## 异常处理
- **[强制]** 确认异常处理后后续逻辑仍能正确执行（变量已赋值）
- **[强制]** 在 raise 异常前记录 error 级别日志

## 资源管理
- **[强制]** 在 finally/defer/with 中关闭数据库/文件/网络连接
- **[建议]** 避免在循环内创建数据库连接/HTTP 客户端等重对象
- **[建议]** 评估共享资源并发访问风险并明确是否需要加锁
- **[建议]** 确保缓存 key 区分不同租户/用户，避免数据串读

## 数据查询与性能
- **[强制]** 为分页查询设置合理上限 `LIMIT`，防止全量拉取
- **[建议]** 优先使用批量操作替代逐条操作，避免 N+1 查询
- **[建议]** 将循环内的数据库/HTTP 查询移到循环外或改为批量接口
- **[建议]** 提取循环内不变的计算/查询到循环外
- **[建议]** 并行化可独立执行的操作，避免串行处理
- **[建议]** 控制对外部服务请求频率，避免高频轮询
- **[建议]** 在集合判断数据量大时用 `set` 替代 `list` 提升性能

## 场景遗漏检查
- **[建议]** 检查修改代码时关联模块是否需同步修改（SDK、portal、配置）
- **[建议]** 确认并发场景是否存在多用户/多线程同时操作同一资源的风险
- **[建议]** 确认操作顺序依赖，确保前置操作完成后再执行依赖操作
- **[建议]** 在多租户场景确认逻辑区分 admin 与普通租户的权限差异
- **[建议]** 考虑调用其他服务时的低版本兼容性

## 接口设计
- **[建议]** 考虑新增 API 参数的向后兼容性并提供默认值
- **[建议]** 为分页接口明确排序字段并设定默认排序
- **[建议]** 明确分层职责，常量不放 datamodels，脱敏逻辑放 controller 层
- **[建议]** 遵循 RESTful 语义：创建用 POST，更新用 PUT，查询用 GET

## 错误码与错误信息
- **[强制]** 确认新增错误码不与已有冲突，避免错误码重复
- **[建议]** 为用户可见错误提供可操作性说明，包含原因和处置建议
- **[建议]** 避免使用模糊通用错误信息，应包含具体失败原因
- **[建议]** 为不同错误场景使用不同错误码，避免共用通用错误码

## 代码质量
- **[建议]** 逐行检查复制粘贴的代码，确保已替换旧逻辑
- **[建议]** 当条件判断超过 3 层嵌套时重构以提高可读性
- **[建议]** 最小化函数入参，内部可查数据不要由调用方传入
- **[建议]** 确保同类操作走统一执行路径，禁止绕过中间层直接操作
- **[建议]** 删除不使用的代码/导入/函数/属性，不要注释保留

## 注释
- **[建议]** 对关键业务逻辑和非直觉实现注释说明为什么
- **[建议]** 为复杂的条件组合添加注释解释原因
- **[建议]** 为临时方案/hack 添加 TODO 注释并标注负责人和预期修复时间

## 容错与降级
- **[建议]** 设计关键流程的异常兜底机制，确保流程状态可追踪
- **[建议]** 提供缓存或外部服务不可用时的降级策略（如回源数据库）
- **[建议]** 设置超时时间不超过执行间隔，并考虑配置变更的低版本兼容

## 空值与边界防护
- **[强制]** 在使用从外部获取的对象/字段前进行 None/空值判断
- **[建议]** 检查可能为 None 的函数返回值后再使用

## 安全
- **[建议]** 确认新增 API 接口的权限校验并检查角色 labels 是否正确

## 国际化
- **[建议]** 使用 i18n 函数包装用户可见文本，避免直接写中文字符串
