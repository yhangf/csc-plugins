# Test-Driven Development — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

先写测试，看它失败，再写最少代码使其通过。

**核心原则：** 如果没有看到测试失败，就不知道它是否在测对的东西。

## 铁律

```
没有先失败的测试，就不能写生产代码
```

先写了代码？删掉，从头开始。不保留作"参考"，不"改编"，不看它。删除即删除。

## 适用场景

**总是使用：** 新功能、Bug 修复、重构、行为变更

**例外（需问用户）：** 一次性原型、生成代码、配置文件

## Red-Green-Refactor 循环

### RED — 写失败测试

写一个最小测试展示期望行为。要求：
- 一个行为
- 清晰命名
- 真实代码（除非不可避免才用 mock）

### 验证 RED — 看它失败（强制，不可跳过）

确认：测试失败（非报错）、失败消息符合预期、因功能缺失而失败（非拼写错误）

### GREEN — 最少代码

写最简单的代码通过测试。不加功能、不重构其他代码、不"改进"。

### 验证 GREEN — 看它通过（强制）

确认：测试通过、其他测试仍通过、输出干净（无错误警告）

### REFACTOR — 清理

仅在 GREEN 之后：消除重复、改进命名、提取辅助函数。保持测试绿色。不加行为。

## 好测试的标准

| 质量 | 好 | 坏 |
|------|---|---|
| 最小化 | 测一件事。名称中有"和"？拆分。 | `test('验证邮箱和域名和空格')` |
| 清晰 | 名称描述行为 | `test('test1')` |
| 展示意图 | 展示期望的 API | 模糊代码应该做什么 |

## 团队测试规范

### 三高优先级测试

聚焦业务关键函数：

| 类型 | 说明 |
|------|------|
| 高频 | 生产环境调用最频繁的函数 |
| 高风险 | 失败会导致数据丢失、安全漏洞或服务中断 |
| 高复杂度 | 圈复杂度 > 10 或多代码路径 |

三高函数**必须**有覆盖所有异常场景的全面测试。

### 断言要求

每个测试**必须**包含有意义的断言：

1. **返回值断言** — 验证函数返回预期值
2. **输出参数断言** — 验证对传入参数的副作用
3. **Mock 验证** — 使用 mock 时验证调用正确

**无断言测试是红线违规。** 没有断言的测试什么也没证明。

### 4 条测试原则

1. 测行为，不测实现
2. 一个测试一个概念
3. 测试独立，不依赖执行顺序
4. 测试要快（毫秒级）

## 常见合理化借口

| 借口 | 现实 |
|------|------|
| "太简单不需要测" | 简单代码也会出 bug，测试只要 30 秒 |
| "之后再测" | 测试立即通过说明什么都没验证 |
| "手动测过了" | 临时测试不等于系统化验证 |
| "TDD 会拖慢我" | TDD 比调试快，测试先行才是高效 |
| "删掉几小时的工作太浪费" | 沉没成本谬误，未验证代码是技术债 |

## 调试集成

发现 bug？先写复现它的失败测试，然后走 TDD 循环。测试证明修复有效并防止回归。

## 唯一规则

```
生产代码 -> 测试存在且先失败过
否则 -> 不是 TDD
```
