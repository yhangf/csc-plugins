# Sysreq Evaluator

## Skill 标识

- Skill name: `sysreq-evaluator`
- Plugin: `cospowers-requirements`
- Scope: Independent split plugin package `cospowers-requirements-plugin`
- Entry skill: `requirements-intake`


**Skill 标识**: `sysreq-evaluator`

其他 skill 通过 `sysreq-evaluator` 引用本 skill。

## Overview

Evaluate system requirements documents (`*-system-requirements.md`) produced by the `system-requirement-analysis` step using **item-by-item rule scanning** (one rule clause at a time, with real-time output), **weighted scoring**, and a **detailed issue report** that pinpoints exactly what is wrong, where, and why.

This skill is a **single-document evaluator** focused exclusively on system requirements documents. Cross-document consistency checks (system requirements ↔ system design ↔ subsystem design) are the responsibility of `doc-quality-evaluator` and are **not performed here**.

**Evaluation order (never skip or reorder):**

```
Phase 0: 加载 → Phase 1: 逐条规范扫描（REQ-01~38） → Phase 1.4: 红线检查 → Phase 1.5: AI-Native 质量筛查（AIN-01~16） → Phase 1.6: 功能需求维度检查（FN-01~04） → Phase 1.7: 交互设计检查（IA-01~05） → Phase 1.8: 补充质量检查（SUP-01~09） → Phase 2: 跳过（跨文档一致性由 doc-quality-evaluator 承担） → Phase 3: 打分 → Phase 4: 生成报告 → Phase 5: 报告保存
```

**检查层次说明：**

| 层次 | 范围 | 说明 |
|---|---|---|
| **核心检查** | REQ-01 至 REQ-38 | 基于模板章节目录的文档完整性评估，共 38 项，直接决定 5 维评分 |
| **红线检查** | R1/R2/R3 | 章节完整性、可靠性、安全性 — 一票否决，任何一项违反总评直接 F |
| **AI-Native 筛查** | AIN-01 至 AIN-16 | AI/自动化流程适配度，16 项，不计入 5 维得分，High 级问题标记阻断风险 |
| **功能需求维度** | FN-01 至 FN-04 | 需求遗漏、细化遗漏、考虑不充分、可实现性，计入需求完整性维度 |
| **交互设计** | IA-01 至 IA-05 | UI/UX 设计质量，计入架构合理性和规范符合性维度 |
| **补充质量** | SUP-01 至 SUP-09 | NFV/OneTrack/国际化/升级回滚等专属检查，计入技术可行性维度 |

## ⚠️ 五大强约束（必须100%严格遵守）

**违反任何一条强约束都是严重错误！**

### 1. 逐条扫描约束：每条规则必须独立检查 ⭐⭐⭐
- ✅ **必须**：对每条规则/检查项，先输出"📍 检查规则 [ID]: [规则内容]"，再执行检查，再输出结论
- ❌ **禁止**：批量跳过多条规则，或只输出"整体符合规范"而不逐条检查
- ✅ **必须**：有问题时，记录文档章节、引用原文、违反的具体规则条款
- ❌ **禁止**：报告模糊描述如"设计不够详细"而不指向具体位置

### 2. 打分约束：分数必须有依据 ⭐⭐⭐
- ✅ **必须**：每个维度的得分附有扣分明细（哪条规则扣了几分）
- ❌ **禁止**：直接给出分数而不说明扣分原因
- ✅ **必须**：红线违反时，总评直接为 F（不论其他维度分数）

### 3. 只报告真实问题：通过的规则不写入报告 ⭐⭐⭐
- ✅ **必须**：只报告真正违反规范或存在不一致的问题
- ❌ **禁止**：写入"符合规范，请保持"等无问题内容
- ✅ **必须**：每个问题都有：位置 + 引用原文 + 违反规则 + 严重级别 + 修改建议

### 4. 一问题一类问题约束：发现一个错误就扫描全文同类 ⭐⭐⭐
- ✅ **必须**：发现某类问题（如"模糊词"）后，立刻扫描全文所有同类实例
- ❌ **禁止**：只记录第一个发现的问题，漏掉同一文档中的其他同类问题

### 5. 全量检查约束：REQ-01 至 REQ-38、AIN/FN/IA/SUP 必须逐条执行 ⭐⭐⭐
- ✅ **必须**：所有检查项（核心 38 条 + AIN 16 条 + FN 4 条 + IA 5 条 + SUP 9 条 + 红线 3 条 = 共 75 条）全部检查，不得跳过任何一条
- ❌ **禁止**：以"文档整体质量良好"为由跳过部分检查项

### 0.1 自动发现文档（Document Discovery）

**不等待用户提供路径。** 按以下顺序自动搜索 `*-system-requirements.md`，找到后直接开始评估。

#### 搜索策略（按优先级依次执行）

**Step A — 搜索标准输出目录**

使用 Glob 在以下路径搜索：
```
docs/agent-rules/specs/**/*-system-requirements.md
```

**Step B — 搜索用户指定的上下文路径**

如果用户在对话中提到了某个目录或文件名，也在该路径下执行上述 Glob 搜索。

**Step C — 向上递归搜索（兜底）**

如果 Step A/B 均未找到任何文档，从当前工作目录向上最多 3 层执行搜索：
```
./**/*-system-requirements.md
```

#### 发现结果处理

| 情况 | 处理方式 |
|---|---|
| 发现 1 份文档 | 直接开始评估，不询问用户 |
| 发现多份文档 | 按修改时间降序排列，建议最新一份，等待用户确认 |
| 用户对话中已直接提供路径 | 跳过 Glob 搜索，直接使用提供的路径 |
| 所有路径均未找到 | 提示"未找到任何系统需求文档，请提供文件路径"，等待用户输入 |

## Phase 1: 逐条规范扫描

对加载的每条检查项，执行以下固定流程：

### 1.1 实时扫描输出格式

对每一条检查项，必须输出：

```
📍 检查 [CHECK-ID]: [规则摘要（来自规范文件的原始描述）]
  ├─ 检查位置: [文档章节 § X.X 或 "全文"]
  ├─ 引用原文: "[被检查内容的直接引用，如果不适用写 N/A]"
  ├─ 规范要求: [该规则的具体要求]
  ├─ 检查结果: ✅ 通过 / ❌ 违规 / ⚠️ 警告
  └─ [仅违规时] 问题ID: [ISSUE-XXX] | 级别: [Critical/Error/Warning] | 扣分: [-N分]
```

### 1.2 违规记录格式

每个发现的问题记录到问题列表：

```
[ISSUE-XXX]
  文档: [文档名]
  位置: § [章节编号] [章节名]
  规则: [CHECK-ID] — [规则描述]
  引用原文: "[实际文档中的原文，精确到句子级]"
  问题描述: [具体说明哪里违反了规则]
  严重级别: Critical / Error / Warning
  修改建议: [具体可操作的修改方向]
  扣分: [-N分] (维度: [维度名])
```

### 1.3 扫描完成统计

扫描完成后输出：

```
📊 [文档名] 扫描完成
  - 扫描规则数: [N] 条
  - 通过: [n] 条
  - 违规(Critical): [n] 条
  - 违规(Error): [n] 条
  - 警告(Warning): [n] 条
  - 发现问题: [ISSUE-001 至 ISSUE-XXX]
```

### 1.4 三条设计红线检查（一票否决）

在规范扫描中，以下三条红线**必须优先检查**，任一违反立刻标记红线，继续扫描但最终总评直接为 F：

**红线 R1 — 章节完整性**

对照 `templates/system-requirement-template.md`，系统需求文档共 9 个一级章节：

Ch.1 文档说明 / Ch.2 关键应用场景 / Ch.3 功能性需求 / Ch.4 非功能性需求（DFX）（含§4.8 日志与可观测性）/
Ch.5 关键测试点 / Ch.6 平台约束需求 / Ch.7 设计任务书 / Ch.8 需求优先级说明 / Ch.9 修订记录

缺失章节数 ≥ 3 → 红线违反。记录：缺失了哪些章节（具体章节号和名称）。

**红线 R2 — 可靠性章节**
- 必须存在且有实质内容（不能只写"不涉及"或"参见平台"）
- 记录：可靠性章节内容是否覆盖：故障容错分析 / 资源使用分析 / 业务流程可靠性

**红线 R3 — 安全性章节**
- 必须存在且有实质内容
- 记录：安全章节内容是否覆盖：安全基线 / 威胁建模 / 数据保护

红线检查输出格式：
```
🚨 红线检查结果:
  R1 章节完整性: ✅ 通过 / ❌ 违反 — 缺失章节: [§X, §Y]
  R2 可靠性章节: ✅ 通过 / ❌ 违反 — 原因: [...]
  R3 安全性章节: ✅ 通过 / ❌ 违反 — 原因: [...]
```

## Phase 1.5: AI-Native Quality Screening（AIN-01 至 AIN-16）

> AI-Native quality screening — identifies structural defects that affect AI/automation workflow usage. Issues in this phase are **supplementary** and do **not** affect the 5-dimension score, but High-level issues are flagged as blocking risks in the report.
>
> **Phase order is mandatory**: Phase 1 (REQ-01~38) → Phase 1.4 (Red Lines) → Phase 1.5 (AIN) → Phase 1.6 (FN/IA)

**输出格式**:
```
📍 AIN-XX: [检查项名称]
  ├─ 检查位置: [§章节/段落]
  ├─ 发现: [通过描述 / 具体问题]
  └─ 结果: ✅ 通过 / ⚠️ [ISSUE-AXX] — 级别: High/Medium/Low
```

### 【内容质量】

#### AIN-01 目标定位明确

文档开头是否显式说明：类型与目的、解决的问题、面向用户、在流程中的位置、支撑的后续动作。
- 检查位置：§高密度摘要、§1.1 目的
- 级别：High

#### AIN-02 范围与边界清楚

是否明确写出"做什么"和"不做什么"；高密度摘要处是否有"不在本次范围内"并明确列出排除项。
- 检查位置：§高密度摘要
- 级别：High

#### AIN-03 约束与禁止项显式化

命名规范、状态定义、权限规则、数据来源、错误处理、安全限制、性能底线、技术栈约束是否全部写出；禁止项是否在高密度摘要"全局强约束"或 §4 DFX 中单独列出。
- 检查位置：§高密度摘要"全局强约束"、§4 各 DFX 小节
- 级别：High

#### AIN-04 优先级明确

需求清单是否有 P0/P1/P2 优先级标记；约束强度是否使用**必须**/**应当**/**可以**/**暂不要求**四级标记。
- 检查位置：§3.1 需求点清单、§4 约束描述
- 级别：Medium

#### AIN-05 具备流程意识

文档是否能回答：前置条件（输入文档）、核心规则（需求内容）、预期产出（输出文档）；是否包含完成定义和验收方式。
- 检查位置：§文档元信息（输入/输出文档）、§3.2 验收标准
- 级别：Medium

### 【表达质量】

#### AIN-06 结构分层

是否有稳定、可扫描的 Markdown 层级（`#` 到 `####`）；文档是否超过 500 行（若超过须检查是否已拆分为主入口+子文档）；章节编号是否连贯无缺失。
- 级别：Medium

#### AIN-07 语义精确

所有性能/可靠性/安全指标是否量化（如 `p99 < 500ms`）；是否存在"合理""适当""性能较好""易用""尽量"等模糊表述。全文扫描所有 DFX 指标。
- 检查位置：§4 所有 DFX 章节、§3.2 约束条件
- 级别：High（每处模糊指标计一个问题）

#### AIN-08 术语一致

关键术语是否在 §1.2 统一定义；全文同一概念是否只有一个名称；是否存在同义词混用（如"租户"/"用户组"/"组织"交替使用）。
- 检查位置：§1.2 定义和缩写，全文术语
- 级别：Medium

#### AIN-09 上下文自洽

关键规则是否集中（非散落各处）；同一需求的口径是否只有一处主入口；例外情况是否单独归类而非夹杂在正文段落中。
- 级别：Medium

#### AIN-10 机器可解析

是否使用 Markdown 标题建立稳定层级；枚举信息是否用列表、键值信息是否用表格；文档头部是否有标准 YAML frontmatter（见 AIN-15）；段落是否一段一主题。
- 级别：Medium

#### AIN-11 自包含与可引用

每节主题是否单一；跨章节引用是否使用明确标题或锚点（如 `§4.1`）；关键术语在首次出现时是否有最小必要定义。
- 级别：Low

#### AIN-12 抗上下文衰减

文档开头是否有高密度摘要，包含：文档目标、适用范围、全局约束、关键禁止项、最终产出；全局强约束是否在对应章节就近重复锚定。
- 检查位置：§高密度摘要
- 级别：High

#### AIN-13 样例充分

异常场景表格是否有具体触发条件和期望行为（非空行）；约束条件是否有量化示例（如"≤ 1000 条"而非"不宜过多"）；UI 约束是否有合法值+非法值示例。
- 检查位置：§3.2.x.3 异常场景、§3.2.x.4 约束条件
- 级别：Medium

#### AIN-14 可验证

每条需求的验收标准是否可用 ✅/❌ 客观判定；是否存在"做好""优化""较好"等无法客观判断是否达成的表述。全文扫描所有验收标准。
- 检查位置：§3.2.x.6 验收标准
- 级别：High

### 【维护质量】

#### AIN-15 版本可追踪

文档头部是否包含标准 YAML frontmatter（必填字段：`name`/`description`/`phase`/`type`/`stage`/`status`/`version`/`updated`）；§9 修订记录是否注明影响范围。
- 级别：Medium

#### AIN-16 可复用

文档结构是否遵循标准模板槽位；平台共性约束（如操作审计、多租户）是否引用标准定义而非重复撰写。
- 级别：Low

## Phase 1.6: Functional Requirement Dimension Checks（FN-01 至 FN-04）

> Supplementary functional requirement quality checks from the V2.0 three-dimension evaluation (functional requirements dimension). These assess user-scenario-to-REQ coverage and requirement quality from a user-need perspective. Issues here map to the **需求完整性** scoring dimension.

#### FN-01 需求遗漏检查

**标准**: 每个用户需求/用户场景均有对应 REQ-XXX 覆盖，无遗漏。

**输出格式**:
```
📍 FN-01: 需求遗漏检查
  ├─ 用户需求/场景数 (§2 + §1.4): [N]
  ├─ REQ 覆盖数 (§3.1): [N]
  ├─ 未覆盖: [列出遗漏的用户需求]
  └─ 结果: ✅ 通过 / ❌ [ISSUE-XXX] — 级别: High | 扣分: -N
```

#### FN-02 需求细化遗漏

**标准**: 功能需求细化不清晰明确；或未覆盖用户场景、操作步骤及期望结果；或遗漏异常场景、异常结果描述。

逐 REQ 检查，每条 REQ 详细说明须覆盖：
- 用户场景描述（§3.2.x.1 功能描述）
- 操作步骤（§3.2.x.2 正常场景）
- 期望结果（§3.2.x.2 正常场景后置状态）
- 异常场景描述（§3.2.x.3 异常场景）
- 异常处理结果（§3.2.x.3 期望系统行为）

缺少任一项记为 Medium 问题；缺少异常场景记为 High 问题。

#### FN-03 需求考虑不充分

**标准**:
- 未考虑关联模块影响（§3.2.x.5 对其他产品或组件的影响为空）
- 缺少针对老功能业务处理逻辑改动点的说明
- 缺少功能特性对资源、兼容性的清晰定义（§3.2.x.4 约束条件）
- 级别：Medium

#### FN-04 需求可实现性未确认

**标准**: 需求未充分与架构师或模块负责人进行技术方案沟通，需求没有可实现性；或需求方案非最佳。检查 §3.2.x.5 依赖与风险中是否有可实现性确认记录。检查 §3.2 技术核实结论表格（REQ-37）。
- 级别：High

## Phase 3: 打分

### 3.1 评分维度与权重

| 维度 | 权重 | 说明 |
|---|---|---|
| **需求完整性** | 35% | 所有 REQ 条目完整，场景覆盖充分，验收条件可验证 — 权重最高因为需求遗漏是系统需求文档的核心风险。**FN-01 至 FN-04 检查结果计入此维度** |
| **技术可行性** | 25% | 性能/可靠性/安全等 DFX 指标有量化依据，约束条件现实可达 |
| **可测试性** | 20% | 每个功能流程覆盖正常路径、异常路径、边界条件三类场景；测试点与需求对齐 |
| **规范符合性** | 15% | 文档格式、章节完整性、命名规范等符合团队规范 |
| **架构合理性** | 5% | 需求边界清晰，约束条件合理，不越权定义设计实现细节 |

> **AIN 问题不计入 5 维得分**，但 High 级 AIN 问题在报告中标记为阻断风险。IA 问题根据内容映射到对应维度（IA-01/IA-02 → 架构合理性；IA-03/IA-04/IA-05 → 规范符合性）。SUP 问题计入技术可行性。

### 3.2 每维度扣分规则

每个维度满分 100 分，按问题严重级别扣分：

| 问题级别 | 说明 | 每个问题扣分 |
|---|---|---|
| **Critical** | 红线违反、需求根本性缺失、安全漏洞 | -40 分 |
| **Error** | 需求未覆盖、逻辑错误、数字不一致、空章节 | -20 分 |
| **Warning** | 命名不规范、建议性问题、数字无来源 | -8 分 |

每个维度最低 0 分（不出现负数）。

**严格扣分示意：**
- 1 个 Critical → 维度直接跌至 60 分以下（D 区）
- 2 个 Error → 维度跌至 60 分（勉强及格线）
- 3 个 Warning → 维度扣 24 分

### 3.3 总分计算

```
总分 = 需求完整性×0.35 + 技术可行性×0.25 + 可测试性×0.20
       + 规范符合性×0.15 + 架构合理性×0.05
```

| 总分 | 等级 | 含义 |
|---|---|---|
| 95-100 | **A** | 优秀，可进入设计阶段 |
| 80-94 | **B** | 良好，必须修复所有 Error 级问题后才能进入设计 |
| 65-79 | **C** | 勉强，必须修复所有 Error 及 Critical 问题并重新评估 |
| <65 | **D** | 不及格，需大幅返工后重新评估 |
| 红线违反 | **F** | 直接失败，不论其他分数 |

### 3.4 打分输出格式

```
📊 评分明细:

┌─────────────────────┬──────┬──────────┬────────────────────────┐
│ 评分维度            │ 权重 │ 维度得分 │ 扣分明细               │
├─────────────────────┼──────┼──────────┼────────────────────────┤
│ 需求完整性          │ 35%  │ [N]/100  │ ISSUE-001(-20), ...    │
│ 技术可行性          │ 25%  │ [N]/100  │ ISSUE-007(-20), ...    │
│ 可测试性            │ 20%  │ [N]/100  │ ISSUE-012(-8), ...     │
│ 规范符合性          │ 15%  │ [N]/100  │ ISSUE-015(-8), ...     │
│ 架构合理性          │  5%  │ [N]/100  │ -                      │
├─────────────────────┼──────┼──────────┼────────────────────────┤
│ **加权总分**        │ 100% │ **[N]**  │                        │
└─────────────────────┴──────┴──────────┴────────────────────────┘
```

## 与其他 Skills 的集成

`sysreq-evaluator` 是一个**被动评估器**，不主动路由下一步。

- 由 `system-requirement-analysis` 步骤 14 以隔离子代理方式派遣
- 评估完成后，将以下内容返回给**调用方 skill**：
  - 评级（A/B/C/D/F）和总分
  - 红线状态（R1/R2/R3）
  - AIN 质量问题（High/Medium/Low 计数及详情）
  - Critical + Error 问题表格（位置、问题描述、修改建议）
  - FN/IA/SUP 补充检查结果
  - 质量报告保存路径
- **不触发任何下一步**；由调用方（`system-requirement-analysis`）根据评级决定：
  - A/B（≥ 80）：调用方继续其自身的下一步（由调用方 skill 的 Checklist 决定，与本 skill 无关）
  - C/D/F：调用方修复问题后重新派遣本 skill，直到评级达到 B 或以上
- 评估报告路径：`docs/agent-rules/specs/quality-reports/YYYY-MM-DD-xxx-quality-report.md`
