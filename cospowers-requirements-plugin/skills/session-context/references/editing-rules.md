# 记忆编辑规则

你是会话编辑，不是记录员。记录员只会往后追加，编辑会审查全局、合并重复、修正过期、删除废弃。

## 核心编辑原则

| 原则 | 规则 | 反例 |
|------|------|------|
| **合并优于追加** | 旧结论被推翻时 **改旧条目**，不要加新条目 | "再加一条说上次那个不对" |
| **删除优于保留** | 已解决的临时假设、已过期的决策、已完成的待办 → **删掉** | 知识库不是历史档案 |
| **精确优于冗长** | 一个条目说清楚一个事实或决策，别塞三个 | 一个文件讲三个不相关的事 |
| **绝对时间** | 永远用 `2026-04-29`，不写 "今天""最近""上周" | "最近改的" → `2026-04-28 修改` |
| **受众不混** | active-*.md 不抄 memory 的全文，memory 不写 active-*.md 的进度 | memory/feedback 里写"当前正在做..." |
| **一条一事** | 一个 memory 文件围绕一个主题，一个条目只讲一个事实 | memory/project/auth.md 里同时讲认证、路由、部署 |

## 三类知识的受众区分

| 位置 | 受众 | 记录什么 | 不记录什么 |
|------|------|---------|-----------|
| **active-*.md** | 下一个 AI（续做任务） | 当前任务目标、进度、阻碍、最近上下文 | 项目级经验、跨任务知识 |
| **memory/feedback/** | 所有未来 AI | 用户纠正过的错误、非显而易见的做法、用户偏好 | 一次性的任务进度 |
| **memory/project/** | 所有未来 AI | 代码库发现、架构决策和原因、模块运作方式 | 用户个人偏好 |

## memory 文件格式约束

```yaml
---
name: <语义化名称，用英文>
description: <一句话描述，用于检索匹配>
type: feedback  # 或 project
---

## 事实记录

### 2026-04-29: <事实标题>

**触发场景**: <什么情况下需要查阅这条>
**结论**: <核心事实或决策>
**原因**: <为什么这样>
```

## 常见编辑场景

| 情况 | 处理方式 | 示例 |
|------|---------|------|
| 过期事实 | 改原条目，更新日期和结论 | 旧："模块 X 用 MySQL" → 新："2026-04-29: 模块 X 已迁移到 PostgreSQL" |
| 相对时间 | 全部转绝对日期 | "今天修复的" → "2026-04-29 修复" |
| 重复记录 | 合并为一条，保留最完整的 | 三条都说"认证用 JWT" → 合并为一条 |
| 推翻的决策 | 删除旧条目，留新决策（不保留历史） | "决定用 Redis 缓存"已被推翻 → 删除，改写为新方案 |
| 跨任务只用一次的临时上下文 | 不放 memory，放 active-*.md | "当前排查的临时假设：可能是网络问题" |
| 记忆间矛盾 | 合并修正为一致结论，无法判断时列到 "未处理" 给用户决定 | feedback 说用方案 A，project 说用方案 B |

## 自检清单（任务结束仪式时必须逐项过）

每次 "沉淀" 操作后逐条验证。哪条打不了勾回去补。

- [ ] active-*.md 中所有可跨任务复用的经验已沉淀到 memory
- [ ] memory 之间没有互相矛盾
- [ ] memory 中没有相对时间（`grep -E "今天\|昨天\|刚刚\|最近\|上周\|today\|yesterday\|recently" memory/` 清零）
- [ ] 所有 memory 文件的 `description` 与内容对得上
- [ ] `MEMORY.md` 索引中的每个链接指向存在的文件
- [ ] 没有两个 memory 文件在说同一件事（重复检查）
- [ ] 新增的 memory 文件已在 `MEMORY.md` 中建立索引
- [ ] active-*.md 已在沉淀完成后清空
