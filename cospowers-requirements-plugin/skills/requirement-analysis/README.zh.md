# Requirement Analysis — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

将原始产品需求转化为结构化、可追溯的 AI 需求文档，遵循 Epic → Feature → Story 三级层次，并承接 TR1 用户需求质量标准（8 条 QS 必过项）。适用于企业级/多服务项目的正式需求文档阶段。

**核心原则：** 用户视角的语言、可测试的验收标准、从业务价值到用户故事的完整追溯、客户验证 ≥ 3、DFX 基线全量落地。

本 skill 内置了需求澄清能力（模糊需求确认、假设识别、范围边界确认），不需要额外经过 brainstorming 阶段。

## 适用场景

- 用户提供原始功能描述、PRD 或口头需求
- 企业或多服务项目需要正式需求文档
- 端到端团队项目流程的起始阶段（在 system-requirement-analysis 之前）

**不适用：** 需求已结构化、创新原型项目、简单单功能任务。

## 15 步检查清单

1. **⛔ GATE — 收集背景情报** — 通过 init 模板收集业务背景、用户画像、场景等，必填字段未完成不得进入步骤 2
2. **查询产品知识库** — 企业项目调用 `kb-query`（中文关键词），查边界/领域/业务流程；**无 kb-query 则跳过，直接 Glob/Grep 代码库理解现有边界**
3. **搜索知识库文档** — 调用 `daedalus-knowledge`；**不可达则跳过，直接使用本地 `rules/` 规范文件**
4. **读取模板** — 加载 `templates/user-requirement-template.md`，内部化其完整结构（TR1 九章结构）
5. **读取原始需求** — 建立初步理解，先不分解
6. **澄清模糊需求** — 一次一个问题，确认范围/术语/假设
7. **确认补充材料** — 主动询问历史测试用例、历史版本、竞品研究等
8. **加载并分级材料** — 少量直接加载；多份先分级（权威/参考/隔离）
9. **分解为 Epic/Feature/Story** — 三级层次，用户视角语言
10. **呈现分解结果供审查** — 展示树结构，获得用户确认
11. **编写 requirements.md** — 输出到 `docs/agent-rules/1-ai-requirements/output/YYYY-MM-DD-<project>-requirements.md`
12. **模板自检** — 逐层检查 Epic/Feature/Story 的必填字段和交接标记
13. **质量评估（关卡）** — 派遣 `aireq-evaluator` 无上下文子代理（`Agent(subagent_type="general-purpose")`），必须达到 B 级（≥ 80 分）。评估 33 项检查（REQ-AI-01~33），含 QS 质量基线 8 项、PP 规划原则、EG 经验指引。不达标则修复后重新派遣
14. **用户审查输出** — 请用户审阅，必要时迭代
15. **转入下一阶段** — 直接调用 `system-requirement-analysis`

## Epic 模板字段（承接 TR1 用户需求质量）

| 字段 | 说明 |
|------|------|
| 【关键价值点】⭐ | 核心业务问题和目标用户价值 |
| 【价值实现程度】⭐ | 可量化的目标，贯穿设计/开发/测试 |
| 【用户需求设计结论】⭐ | 7 项：价值规划、目标用户与现状旅程、核心痛点(量化+来源)、设计策略(意图/目标/创新)、Playbook、竞品对比、注意事项 |
| 【客户验证结论】⭐ | QS-001：≥3 个客户验证记录，不可裁剪 |
| 【质量需求】⭐ | QS-003：P0/P1 问题池（来源标注），不可裁剪 |
| 【包含 Feature】⭐ | 子 Feature 列表 |
| 【跨团队协作】⭐ | QS-002：共识状态或裁剪说明 |
| 【责任人】👤 | 人工填写 |

## Feature 模板字段

| 字段 | 说明 |
|------|------|
| 【关联关键价值点】⭐ | 链接到父 Epic 的价值点 |
| 【特性需求分类】⭐ | 基本可用 / 满足主流体验 / 构建竞争优势，含分类理由 |
| 【功能级 Playbook】⭐ | 用户视角场景链路 |
| 【场景建模】🔲 | 复杂 Feature 填写三级场景表 |
| 【特性验收条件】⭐ | 用户视角的关键效果和指标 |
| 【非功能需求约束 DFX】⭐ | QS-004：覆盖性能/可靠性/安全性/可测试性/可运维性等 6 维 |
| 【包含 Story】⭐ | 子 Story 列表 |

## Story 模板字段

| 字段 | 说明 |
|------|------|
| 【用户故事】⭐ | "作为&lt;角色&gt;，我想要&lt;活动&gt;，以便于&lt;价值&gt;" |
| 【场景说明】⭐ | 目标客户 + 痛点背景（关联 Epic 痛点来源） |
| 【A/C 验收条件】⭐ | 功能性(Given-When-Then 正常+异常) + 非功能性 |
| 【安全需求】⭐ | 必填，即使是"不涉及"也要写明（QS-007） |
| 【海外合规】🔲 | QS-006：海外基线自检或裁剪说明 |
| 【技术思路】👤 | 必须写"待开发负责人填写"，AI 不得虚构 |
| 【工作量评估】👤 | 必须写"待评估" |
| 【交接标记】⭐ | 每个 Story 末尾 🔴/🟡/🟢 |

## 8 条 QS 必过项（承接 TR1 用户需求标准）

| QS | 检查项 | 裁剪 |
|----|--------|------|
| QS-001 | 客户验证 ≥ 3，有据可查 | 不可裁剪 |
| QS-002 | 协作方需求共识 | 无协作需求可裁剪 |
| QS-003 | P0/P1 严重问题纳入需求池 | 不可裁剪 |
| QS-004 | DFX 需求基线全量落地 | 不可裁剪 |
| QS-005 | 单需求含基本内容 | 产线可自定义 |
| QS-006 | 海外需求基线自检 | 不涉外出海可裁剪 |
| QS-007 | 道德合规 TR1 检视 | 不可裁剪 |
| QS-008 | 客户成功基线需求 | 非 XAAS 可裁剪 |

## 补充材料分级规则（多份材料时）

| 等级 | 典型材料 | 处理方式 |
|------|---------|---------|
| **权威** | 主需求文档、客户签字确认的需求 | 直接作为基准 |
| **参考** | 历史设计、竞品分析、技术预研 | 有选择性采纳，需判断过滤 |
| **隔离** | 测试用例、测试工具文档 | 只提取用户意图，不让其结构或命名影响需求 |

## 交接标记

每个 Story 完成后标记状态：🔴 重点细化 / 🟡 待确认 / 🟢 已明确
