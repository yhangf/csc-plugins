# Aireq Evaluator

## Skill 标识

- Skill name: `aireq-evaluator`
- Plugin: `cospowers-requirements`
- Scope: Independent split plugin package `cospowers-requirements-plugin`
- Entry skill: `requirements-intake`


**Skill 标识**: `aireq-evaluator`

其他 skill 通过 `aireq-evaluator` 引用本 skill。

## Overview

Evaluate TR1 user requirements documents (`*-requirements.md`) produced by the `requirement-analysis` step using **item-by-item rule scanning** (one rule clause at a time, with real-time output), **weighted scoring**, and a **detailed issue report** that pinpoints exactly what is wrong, where, and why.

This skill is a **single-document evaluator** focused exclusively on TR1 user requirements documents (9-chapter structure). Cross-document consistency checks (TR1 requirements ↔ system requirements ↔ design) are the responsibility of `doc-quality-evaluator` and are **not performed here**.

**Evaluation order (never skip or reorder):**

```
Phase 0: 加载 → Phase 1: 逐条规范扫描 → Phase 2: 跳过（跨文档一致性由 doc-quality-evaluator 承担） → Phase 3: 打分 → Phase 4: 生成报告
```

## ⚠️ 四大强约束（必须100%严格遵守）

**违反任何一条强约束都是严重错误！**

### 1. 逐条扫描约束：每条规则必须独立检查 ⭐⭐⭐
- ✅ **必须**：对每条规则/检查项，先输出"📍 检查规则 [ID]: [规则内容]"，再执行检查，再输出结论
- ❌ **禁止**：批量跳过多条规则，或只输出"整体符合规范"而不逐条检查
- ✅ **必须**：有问题时，记录文档章节、引用原文、违反的具体规则条款
- ❌ **禁止**：报告模糊描述如"内容不够详细"而不指向具体位置

### 2. 只报告真实问题：通过的规则不写入报告 ⭐⭐⭐
- ✅ **必须**：只报告真正违反规范或存在不一致的问题
- ❌ **禁止**：写入"符合规范，请保持"等无问题内容
- ✅ **必须**：每个问题都有：位置 + 引用原文 + 违反规则 + 严重级别 + 修改建议

### 3. 一问题一类问题约束：发现一个错误就扫描全文同类 ⭐⭐⭐
- ✅ **必须**：发现某类问题（如"模糊词"）后，立刻扫描全文所有同类实例
- ❌ **禁止**：只记录第一个发现的问题，漏掉同一文档中的其他同类问题

### 4. 全量检查约束：REQ-AI-01 至 REQ-AI-34 必须逐条执行 ⭐⭐⭐
- ✅ **必须**：34 条内置检查项全部检查，不得跳过任何一条
- ❌ **禁止**：以"文档整体质量良好"为由跳过部分检查项
- ✅ **必须**：结构性问题（Error/Critical）阻塞移交；建议性问题（Warning）不阻塞

## Phase 0: 加载阶段

### 0.1 自动发现文档（Document Discovery）

**不等待用户提供路径。** 按以下顺序自动搜索 `*-requirements.md`，找到后直接开始评估。

#### 搜索策略（按优先级依次执行）

**Step A — 搜索标准输出目录**

使用 Glob 在以下路径搜索：
```
docs/agent-rules/1-ai-requirements/output/*-requirements.md
```

**Step B — 搜索用户指定的上下文路径**

如果用户在对话中提到了某个目录或文件名，也在该路径下执行上述 Glob 搜索。

**Step C — 向上递归搜索（兜底）**

如果 Step A/B 均未找到任何文档，从当前工作目录向上最多 3 层执行搜索：
```
./**/*-requirements.md
```

#### 发现结果处理

| 情况 | 处理方式 |
|---|---|
| 发现 1 份文档 | 直接开始评估，不询问用户 |
| 发现多份文档 | 按修改时间降序排列，建议最新一份，等待用户确认 |
| 用户对话中已直接提供路径 | 跳过 Glob 搜索，直接使用提供的路径 |
| 所有路径均未找到 | 提示"未找到任何 TR1 用户需求文档，请提供文件路径"，等待用户输入 |

找到文档后输出：

```
📄 发现 TR1 用户需求文档，开始评估：
  📋 docs/agent-rules/1-ai-requirements/output/YYYY-MM-DD-xxx-requirements.md

📂 评估模式: 单文档（TR1 用户需求专项评估）
⚠️ 说明: 跨文档一致性核查由 doc-quality-evaluator 承担，本 skill 不执行

开始评估...
```

### 0.1 文档定位

**目的**：确认待评审文档是 TR1 用户需求规格说明书，而非其他阶段文档。

**执行步骤**：
1. 检查文档 YAML frontmatter 的 `stage` 字段是否为 `TR1`
2. 检查文档是否包含 TR1 九章结构：详细用户需求规格(§1)、客户验证结论、协作方需求共识、质量需求、非功能性需求(§2)、XAAS 客户成功(§3)、附录、文档完成自检清单
3. **排除**：若文档包含"系统需求规格说明书"、"总体设计"、"概要设计"、"模块设计"等 TR2 及以后阶段关键词，停止评审并输出说明

### 0.2 加载评审规范

**读取**以下规范文件（必须实际读取，不可跳过），并使用内置检查项：

```
TR1 用户需求文档评估 → 读取:
  rules/design-review/design-01-doc-writing-standards-checklist.md
  rules/requirement-checklists/ai-native-doc-adaptation-checklist.md

  rules/requirement-checklists/user-requirement-planning-principles-checklist.md

## Phase 1: 逐条规范扫描

对加载的每条检查项，执行以下固定流程：

### 1.1 实时扫描输出格式

对每一条检查项，必须输出：

```
📍 检查 [CHECK-ID]: [规则摘要（来自规范文件的原始描述）]
  ├─ 检查位置: [§章节名称 或 "全文"]
  ├─ 引用原文: "[被检查内容的直接引用，如果不适用写 N/A]"
  ├─ 规范要求: [该规则的具体要求]
  ├─ 检查结果: ✅ 通过 / ❌ 违规 / ⚠️ 警告
  └─ [仅违规时] 问题ID: [ISSUE-XXX] | 级别: [Critical/Error/Warning] | 扣分: [-N分]
```

### 1.2 违规记录格式

每个发现的问题记录到问题列表：

```
[ISSUE-XXX]
  文档: [文档名]
  位置: [§章节名称及层级]
  规则: [CHECK-ID] — [规则描述]
  引用原文: "[实际文档中的原文，精确到句子级]"
  问题描述: [具体说明哪里违反了规则]
  严重级别: Critical / Error / Warning
  修改建议: [具体可操作的修改方向]
  扣分: [-N分] (维度: [维度名])
```

### 1.3 扫描完成统计

扫描完成后输出：

```
📊 [文档名] 扫描完成
  - 扫描规则数: [N] 条
  - 通过: [n] 条
  - 违规(Critical): [n] 条
  - 违规(Error): [n] 条
  - 警告(Warning): [n] 条
  - 发现问题: [ISSUE-001 至 ISSUE-XXX]
```

### 1.4 四条设计红线检查（一票否决）

在规范扫描中，以下四条红线**必须优先检查**，任一违反立刻标记红线，继续扫描但最终总评直接为 F：

**红线 R1 — 根特性结构缺失**

全文扫描是否存在：
- 无 §1.1.x 价值根特性（缺少核心需求章节）
- 根特性缺少 §1.1.1 用户场景分析或 §1.1.4 特性详细需求规格（核心子节缺失）

有根特性结构缺失 → 红线违反。记录：缺失内容及位置。

**红线 R2 — 合规性检视大面积空白**

统计 §2.2.4 合规性的 QS-007 4 类检视项：
- 超过 50% 的检视项为空白（非"通过/不通过/不涉及"，而是真正的空白/未填写）→ 红线违反

记录：空白检视项列表及占比。

**红线 R3 — 未完成占位符**

全文扫描是否含"TBD/TODO/待补充"等未完成标记（§1.1.5 技术需求和 §1.1.4 遗留问题中合理标注的"无"除外）：
- 发现任意未完成占位符 → 红线违反

记录：占位符所在位置和内容。

**红线 R4 — QS 一票否决项**

以下 QS 项为不可裁剪项，任一不通过 → 红线违反：
- RULE-UR-QS-001 客户验证 ≥ 3（REQ-AI-23）
- RULE-UR-QS-003 P0/P1 问题池（REQ-AI-25）
- RULE-UR-QS-004 DFX 需求基线（REQ-AI-26）
- RULE-UR-QS-007 道德合规检视（REQ-AI-29）

记录：触发红线的 QS 项及缺失证据。

红线检查输出格式：
```
🚨 红线检查结果:
  R1 根特性结构缺失: ✅ 通过 / ❌ 违反 — 缺失内容: [详情]
  R2 合规性检视空白: ✅ 通过 / ❌ 违反 — 空白比例: [N%，N个/总数]
  R3 未完成占位符: ✅ 通过 / ❌ 违反 — 位置: [...]
  R4 QS 一票否决: ✅ 通过 / ❌ 违反 — 触发项: [QS-00X ...]
```

## Phase 3: 打分

### 3.1 评分维度与权重

| 维度 | 权重 | 说明 |
|---|---|---|
| **结构完整性** | 20% | TR1 九章齐全，§1.1.x 根特性结构完整，§1.1.4 需求规格完整 |
| **DFX 需求质量** | 20% | 性能量化、可靠性基线、合规性检视、全维度覆盖（REQ-AI-13~16） |
| **QS 质量基线** | 20% | 客户验证、P0/P1 问题池、DFX 基线、道德合规检视（REQ-AI-23~30） |
| **需求规格质量** | 15% | 需求 ID 格式、用户视角、验收标准可测量、痛点可追溯（REQ-AI-06~09） |
| **设计策略质量** | 10% | 设计目标可量化、策略可判断、Playbook 价值导向（REQ-AI-10~12） |
| **移交就绪性** | 10% | 无占位符、技术预研与风险评估、需求可追溯（REQ-AI-20~22） |
| **AIN 文档质量** | 5% | 范围边界、术语一致性、自检清单完成（REQ-AI-17~19） |

### 3.2 每维度扣分规则

每个维度满分 100 分，按问题严重级别扣分：

| 问题级别 | 说明 | 每个问题扣分 |
|---|---|---|
| **Critical** | 红线违反、层级根本性缺失 | -40 分 |
| **Error** | 字段缺失、格式错误、空白安全需求、占位符 | -20 分 |
| **Warning** | 角色不具体、价值模糊、量化缺失 | -8 分 |

每个维度最低 0 分（不出现负数）。

### 3.3 总分计算

总分 = 结构完整性×0.20 + DFX需求质量×0.20 + QS质量基线×0.20
       + 需求规格质量×0.15 + 设计策略质量×0.10 + 移交就绪性×0.10 + AIN文档质量×0.05
       + AIN文档质量×0.15 + 用户故事质量×0.10 + 移交就绪性×0.05
```

| 总分 | 等级 | 含义 |
|---|---|---|
| 95-100 | **A** | 优秀，可进入系统需求分析阶段 |
| 80-94 | **B** | 良好，必须修复所有 Error 级问题后才能移交 |
| 65-79 | **C** | 勉强，必须修复所有 Error 及 Critical 问题并重新评估 |
| <65 | **D** | 不及格，需大幅返工后重新评估 |
| 红线违反 | **F** | 直接失败，不论其他分数 |

## 与其他 Skills 的集成

`aireq-evaluator` 是一个**被动评估器**，不主动路由下一步。

- 由 `requirement-analysis` 步骤 13 以隔离子代理方式派遣
- 评估完成后，将以下内容返回给**调用方 skill**：
  - 评级（A/B/C/D/F）和总分
  - 红线状态（R1/R2/R3/R4）
  - Critical + Error 问题表格（位置、问题描述、修改建议）
  - 质量报告保存路径
- **不触发任何下一步**；由调用方（`requirement-analysis`）根据评级决定：
  - A/B（≥ 80）：调用方继续其自身的下一步（由调用方 skill 的 Checklist 决定，与本 skill 无关）
  - C/D/F：调用方修复问题后重新派遣本 skill，直到评级达到 B 或以上
- 评估报告路径：`docs/agent-rules/1-ai-requirements/output/quality-reports/YYYY-MM-DD-xxx-quality-report.md`
