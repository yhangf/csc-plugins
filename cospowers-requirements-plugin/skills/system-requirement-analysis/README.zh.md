# System Requirement Analysis — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

将 AI 需求（Epic/Feature/Story）深化为正式的系统需求文档，遵循团队标准模板的 9 章结构。每个需求项（REQ-XXX）需包含 5 类场景的 Given-When-Then 格式验收标准。

模板已升级：YAML frontmatter 元数据、§4.8 开放性要求、附录文档完成自检清单。

**核心原则：** 单一结构化文档、所有章节必填（不涉及写原因）、独立可验证、所有场景覆盖。

## 适用场景

- AI 需求已存在，需系统级深化
- 企业项目需要正式系统需求文档
- 端到端流程中 requirement-analysis 的下一阶段

## 15 步检查清单

1. **⛔ GATE — 收集背景情报** — 通过 init 模板收集业务背景、架构图、领域术语等，必填字段未完成不得进入步骤 2
2. **知识归档** — 从 init 模板提取有价值的业务规则、约束、集成模式，调用 `evo-knowledge-wheel` 归档；**无远程服务则自动切换本地 `memory/kb/` 存储**
3. **加载输入需求** — 读取 AI 需求文档（Epic/Feature/Story）
4. **查询产品知识库** — 企业项目调用 `kb-query`（中文关键词）；**无 kb-query 则跳过，直接 Glob/Grep 代码库**
5. **读取模板** — 加载 `templates/system-requirement-template.md`，内部化 9 章结构
6. **搜索知识库文档** — 调用 `daedalus-knowledge`，查询平台 DFX 约束；**不可达则跳过，使用本地 `rules/dfx/` 文件替代**
7. **呈现需求项计划** — 展示 REQ-XXX 列表，获取用户确认
8. **推理需求完整性** — 主动推理遗漏需求（创建→删除/修改、列表→搜索/过滤、权限→审计等）
9. **处理调整请求** — 迭代至用户满意
10. **确认生成范围** — 全量 / 核心优先 / 示例驱动
11. **生成系统需求文档** — 遵循 9 章模板（含 YAML frontmatter、高密度摘要、§4.8 开放性要求），保存到 `docs/agent-rules/2-system-requirements/output/`
12. **完整性验证 + 跨章节一致性检查** — 所有章节已填写、5 类一致性通过
13. **子代理评审** — 派遣独立子代理使用 `system-requirement-reviewer-prompt.md` 评审
14. **质量评估（关卡）** — 派遣 `sysreq-evaluator` 无上下文子代理（`Agent(subagent_type="general-purpose")`），75 项检查（REQ-01~38 + AIN-01~16 + FN-01~04 + IA-01~05 + SUP-01~09）+ 5 维加权打分，必须达到 B 级（≥ 80 分）
15. **转入设计阶段** — 调用 `design-spec`

## 文档 9 章结构

| 章节 | 标题 | 关键内容 |
|------|------|---------|
| 高密度摘要 | 全文关键信息前置索引 | 需求范围、REQ 总数/P0-P2 统计、全局强约束、排除项 |
| 1 | 文档说明 | 目的、定义/缩写、参考、版本需求范围（含排除项） |
| 2 | 关键应用场景 | 用户场景（角色、流程、预期结果、业务价值） |
| 3 | 功能性需求 | REQ-XXX 条目（功能描述、正常/异常场景、约束条件、依赖风险、验收标准、当前实现、技术核实） |
| 4 | DFX 要求 | 性能/可靠性/安全性/可运维性/可扩展性/兼容性/可测试性/开放性 |
| 5 | 关键测试点 | 功能/性能/可靠性/安全/兼容性测试点 |
| 6 | 平台约束需求 | 产品约束、开发规范、平台公共能力、环境约束 |
| 7 | 设计任务书 | 需求追溯表、模块目标、DFX 目标、流程要求 |
| 8 | 需求优先级说明 | P0/P1/P2 定义和时间要求 |
| 9 | 修订记录 | 文档修订历史 |
| 附录 | 文档完成自检清单 | 内容完整性、DFX 要求、需求质量、AI-Native 质量、跨文档一致性 |

## sysreq-evaluator 检查项构成（75 项）

| 分类 | 数量 | 范围 |
|------|------|------|
| REQ 基础检查 | 38 项（REQ-01~38） | 9 章结构完整性、验收条件质量、DFX 量化等 |
| AIN AI-Native 质量 | 16 项（AIN-01~16） | 内容质量、表达质量、维护质量。不参与 5 维打分，High 级阻断 |
| FN 功能需求 | 4 项（FN-01~04） | 需求遗漏/细化遗漏/考虑不充分/可实现性 |
| IA 交互设计 | 5 项（IA-01~05） | UI 场景满足/流程清晰/客户导向/原型规范/约束完整 |
| SUP 非功能性补充 | 9 项（SUP-01~09） | 升级/迁移/硬件/第三方安全/测试工具/国际化/NFV/OneTrack |

## 每个 REQ 条目必须覆盖的 5 类场景

| 场景类型 | 要求数量 | 说明 |
|---------|---------|------|
| 正常流程 | 2-4 个 | 主流程的具体数据成功路径 |
| 异常流程 | 2-3 个 | 错误输入、系统故障、网络问题 |
| 安全场景 | 2 个 | 权限验证、数据加密、访问控制 |
| 边界场景 | 2 个 | 极限值、资源上限、并发限制 |
| 可测试场景 | 1-2 个 | 测试数据构造和验证方法 |

## DFX 维度

| 维度 | 关键指标 |
|------|---------|
| 性能要求 | 并发、响应时间(P95/P99)、吞吐量、CPU/内存 |
| 可靠性要求 | 可用性(SLA)、RTO/RPO、MTBF/MTTR、过载保护 |
| 安全性要求 | 认证授权、数据加密/脱敏、审计日志、安全基线检视 |
| 可运维性要求 | 部署方式、监控告警、日志管理、配置管理 |
| 可扩展性要求 | 水平/垂直扩展能力 |
| 兼容性要求 | 浏览器、操作系统、数据库、向后兼容 |
| 可测试性要求 | 接口可测、状态可观测、故障可注入 |
| 开放性要求 | OpenAPI 暴露、认证方式、内部接口隔离 |

## 历史版本处理

| 标签 | 含义 |
|------|------|
| `[延续]` | 与旧版本一致 |
| `[新增]` | 本迭代全新需求 |
| `[变更-来自REQ-XXX]` | 修改自旧版本 |
| `[废弃]` | 旧版本有但本迭代删除 |

## 重要规则

- 所有章节必填，不涉及写"不涉及"及原因
- YAML frontmatter 必须包含，`stage: TR2`
- 高密度摘要必须含全局强约束和排除项
- 附录自检清单必须在提交前逐项检查
- DFX 数字必须来自知识库文档或用户确认，不得使用模板示例值
