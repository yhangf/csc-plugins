# Aireq Evaluator — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

对 `requirement-analysis` 产出的 AI 需求文档（`*-requirements.md`）进行逐条规范扫描和质量打分。单文档评估器，专注 Epic/Feature/Story 三级层次的结构完整性、验收条件质量和移交就绪性。跨文档一致性由 `doc-quality-evaluator` 负责。

## 审查流程

```
Phase 0: 加载（自动发现文档 + 加载规范）
  → Phase 1: 逐条规范扫描（REQ-AI-01 至 REQ-AI-33）
  → Phase 2: 跳过（跨文档一致性由 doc-quality-evaluator 承担）
  → Phase 3: 加权打分
  → Phase 4: 生成评估报告
```

## 检查项构成（33 项）

| 类别 | 编号 | 数量 | 说明 |
|------|------|:--:|------|
| 结构完整性 | REQ-AI-01~04 | 4 | Epic/Feature/Story 三层字段齐全，层级关系正确 |
| 用户故事质量 | REQ-AI-05~08 | 4 | 三段式格式、角色具体性、价值可测量性、痛点对比 |
| 验收条件质量 | REQ-AI-09~13 | 5 | GWT 格式、正常+异常双覆盖、无模糊词、安全需求填写 |
| AIN 文档质量 | REQ-AI-14~18 | 5 | 范围边界、术语一致性、DFX 量化、优先级标记、QS 质量基线 |
| 移交就绪性 | REQ-AI-19~21 | 3 | 无占位符、技术思路不越权、需求可追溯 |
| QS 质量基线 | REQ-AI-22~29 | 8 | 客户验证 ≥3、协作方共识、P0/P1 问题池、DFX 基线、合规检视等 |
| 产品规划质量 | REQ-AI-30~33 | 4 | 战略一致性、功能分类合理、战略承接、关键劣势披露 |

## 评分体系

| 维度 | 权重 | 说明 |
|------|:--:|------|
| 结构完整性 | 25% | Epic/Feature/Story 三层字段齐全，层级关系正确 |
| 验收条件质量 | 25% | GWT 格式、双路径覆盖、无模糊词、安全需求填写 |
| QS 质量基线 | 20% | 客户验证、P0/P1 问题池、DFX 基线、合规检视 |
| AIN 文档质量 | 15% | 范围边界、术语一致性、DFX 量化、优先级标记 |
| 用户故事质量 | 10% | 三段式格式、角色具体性、价值可测量性、痛点对比 |
| 移交就绪性 | 5% | 无占位符、技术思路不越权填写、需求可追溯 |

扣分：Critical -40 / Error -20 / Warning -8。每个维度最低 0 分。

| 总分 | 等级 | 含义 |
|------|:--:|------|
| 95-100 | A | 优秀，可进入系统需求分析 |
| 80-94 | B | 良好，修复 Error 级问题后移交 |
| 65-79 | C | 勉强，修复 Error+Critical 后重新评估 |
| <65 | D | 不及格，大幅返工 |
| 红线违反 | F | 直接失败 |

## 4 条红线（一票否决）

| 红线 | 标准 |
|------|------|
| R1 孤立层级 | 存在孤立 Story（无 Feature 父节点）或孤立 Feature（无 Epic 父节点） |
| R2 安全需求空白 | 超过 50% 的 Story 安全需求字段为空白（非"不涉及"） |
| R3 未完成占位符 | 全文含 TBD/TODO/待补充（技术思路/工作量评估例外） |
| R4 QS 一票否决 | QS-001 客户验证 / QS-003 P0/P1 问题池 / QS-004 DFX 基线 / QS-007 合规检视，任一不可裁剪项不通过 |

## 派遣方式

由 `requirement-analysis` 步骤 13 以隔离子代理（`Agent(subagent_type="general-purpose")`）方式派遣。评估完成后将评级、红线状态、问题表格、报告路径返回给调用方，不主动路由下一步。

## 下一步

- 评级 A/B（≥ 80）→ `requirement-analysis` 继续步骤 14，进入 `system-requirement-analysis`
- 评级 C/D/F → 修复问题后重新派遣 `aireq-evaluator`，直至达到 B 级
