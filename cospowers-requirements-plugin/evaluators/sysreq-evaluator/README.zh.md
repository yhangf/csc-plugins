# Sysreq Evaluator — 中文参考说明

> 本文件仅供人阅读，不加入 AI 上下文。AI 使用的是同目录下的 `SKILL.md`（英文）。

## 概述

对 `system-requirement-analysis` 产出的系统需求文档（`*-system-requirements.md`）进行多层逐条规范扫描和质量打分。单文档评估器，共 75 项检查分 6 个层次执行。跨文档一致性由 `doc-quality-evaluator` 负责。

## 审查流程

```
Phase 0: 加载（自动发现文档 + 加载规范）
  → Phase 1: 逐条规范扫描（REQ-01~38，核心 38 项）
  → Phase 1.4: 红线检查（R1/R2/R3，一票否决）
  → Phase 1.5: AI-Native 质量筛查（AIN-01~16，不计分但标记阻断风险）
  → Phase 1.6: 功能需求维度检查（FN-01~04，计入需求完整性）
  → Phase 1.7: 交互设计检查（IA-01~05，计入架构/规范维度）
  → Phase 1.8: 补充质量检查（SUP-01~09，计入技术可行性）
  → Phase 2: 跳过（跨文档一致性由 doc-quality-evaluator 承担）
  → Phase 3: 加权打分
  → Phase 4: 生成报告
  → Phase 5: 报告保存
```

## 检查项构成（75 项）

| 层次 | 编号 | 数量 | 计入评分 |
|------|------|:--:|------|
| 核心检查 | REQ-01~38 | 38 | 5 维得分 |
| 红线检查 | R1/R2/R3 | 3 | 一票否决 |
| AI-Native 筛查 | AIN-01~16 | 16 | 不计分（High 级标记阻断风险） |
| 功能需求维度 | FN-01~04 | 4 | 需求完整性 |
| 交互设计 | IA-01~05 | 5 | 架构合理性 / 规范符合性 |
| 补充质量 | SUP-01~09 | 9 | 技术可行性 |

核心 38 项覆盖：元信息、目的、术语、应用场景、功能需求清单与详情、性能/可靠性/安全/可运维/可扩展/兼容/可测试 DFX、测试点对齐、平台约束、需求跟踪、修订记录、版本范围等。

## 评分体系

| 维度 | 权重 | 说明 |
|------|:--:|------|
| 需求完整性 | 35% | 所有 REQ 条目完整，场景覆盖充分，验收条件可验证。FN-01~04 计入此维度 |
| 技术可行性 | 25% | DFX 指标有量化依据，约束条件现实可达。SUP-01~09 计入此维度 |
| 可测试性 | 20% | 功能流程覆盖正常/异常/边界三类场景，测试点与需求对齐 |
| 规范符合性 | 15% | 文档格式、章节完整性、命名规范。IA-03/04/05 计入此维度 |
| 架构合理性 | 5% | 需求边界清晰，约束合理。IA-01/02 计入此维度 |

扣分：Critical -40 / Error -20 / Warning -8。AIN 问题不计入 5 维得分。

| 总分 | 等级 | 含义 |
|------|:--:|------|
| 95-100 | A | 优秀，可进入设计阶段 |
| 80-94 | B | 良好，修复 Error 级问题后进入设计 |
| 65-79 | C | 勉强，修复 Error+Critical 后重新评估 |
| <65 | D | 不及格，大幅返工 |
| 红线违反 | F | 直接失败 |

## 3 条红线（一票否决）

| 红线 | 标准 |
|------|------|
| R1 章节完整性 | 缺失章节数 ≥ 3（共 9 个一级章节，含 §4.8 日志与可观测性） |
| R2 可靠性章节 | 必须存在且有实质内容（覆盖故障容错/资源使用/业务流程可靠性） |
| R3 安全性章节 | 必须存在且有实质内容（覆盖安全基线/威胁建模/数据保护） |

## 派遣方式

由 `system-requirement-analysis` 步骤 14 以隔离子代理（`Agent(subagent_type="general-purpose")`）方式派遣。评估完成后将评级、红线状态、AIN 质量问题、问题表格、报告路径返回给调用方，不主动路由下一步。

## 下一步

- 评级 A/B（≥ 80）→ `system-requirement-analysis` 继续下一步，进入 `design-spec`
- 评级 C/D/F → 修复问题后重新派遣 `sysreq-evaluator`，直至达到 B 级
