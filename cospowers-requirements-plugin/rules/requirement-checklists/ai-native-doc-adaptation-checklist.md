# AI-Native 文档通用适配要求

---

## 一、核心定义

AI-Native 文档：**同时作为人、AI、自动化流程三方协作的高质量输入**，必须满足：

1. **人能快速看懂**
2. **AI 能低歧义理解**
3. **后续流程能基于它继续执行、验证、追踪**

目标：**可理解、可执行、可验证、可追踪、可复用**。

> 传统文档重“可读”，AI-Native 文档重“可读 + 可执行”。

---

## 二、与传统文档的核心差异

| 维度 | 传统文档 | AI-Native 文档 |
|------|----------|----------------|
| 主要读者 | 人类团队成员 | 人类 + AI 模型 + 自动化流程 |
| 语言风格 | 自然语言，模糊可接受 | 精确、无歧义、结构化 |
| 容错空间 | 大（人能理解意图） | 极小（AI 按字面执行） |
| 细节补全 | 靠口头沟通 | 必须文字显式化 |
| 更新滞后 | 勉强能用 | 直接导致 AI 输出失准 |

**关键区别**：传统文档的质量下限由“聪明的读者能理解意图”兜底；AI-Native 文档没有这条兜底线——**文档质量直接决定 AI 输出质量的上限**。

---

## 三、核心要求（16 条快速检查）

### 【内容质量】

#### 1. 目标定位明确
- 文档开头显式交代：类型与目的、解决的问题、面向用户、在流程中的位置、支撑的后续动作。

#### 2. 范围与边界清楚
- 写清“做什么”，也写清“不做什么”。
- 显式说明：本次/本期范围、明确排除的范围、不支持的情况、与相邻模块的边界。

#### 3. 约束与禁止项显式化
- 命名规范、状态定义、权限规则、数据来源、错误处理、安全限制、性能底线、技术栈约束等必须全部写出。
- **禁止项**必须单独列出（示例：禁止硬编码配置、禁止在循环内查询数据库）。

#### 4. 优先级明确
- 用四级标记区分约束强度：**必须**（绝对约束）、**应当**（强烈推荐）、**可以**（建议满足）、**暂不要求**（明确后置）。

#### 5. 具备流程意识
- 文档应能回答：需要什么输入（前置条件）、做什么处理（核心规则）、产出什么输出（预期产出）。
- 内置闭环信息：完成定义、验收方式、依赖项、风险项、待确认项。

### 【表达质量】

#### 6. 结构分层
- 必须有稳定、可扫描的层级。长文档（>500 行）必须拆分为 **主入口（TOC）+ 渐进式披露的子文档**。
- 推荐骨架：元信息层 → 背景与目标层 → 范围与边界层 → 术语与对象层 → 核心规则层 → 约束与禁止层 → 输入输出层 → 异常与边界层 → 验证标准层 → 版本与变更层。

#### 7. 语义精确
- **能量化的量化、能列举的列举、能枚举的枚举、能定边界的定边界、能拆成规则的拆成规则**。
- 避免形容词（如“性能较好”），使用具体指标（如“p99 < 500ms”）。

#### 8. 术语一致
- 一个概念只能有一个名称。
- 关键术语、对象命名、状态口径全局统一。
- 在术语表中统一声明并锁定。

#### 9. 上下文自洽
- 关键规则集中，关键结论前置。
- 同一问题的最终口径只有一处主入口。
- 例外情况单独归类，不散落在正文各处。

#### 10. 机器可解析
- 使用 Markdown 标题建立稳定层级。
- 枚举信息优先用列表，键值信息优先用表格、YAML、JSON 或代码块。
- 一段只表达一个主题。
- 文档头部提供标准元信息（示例见下方）。

#### 11. 自包含与可引用
- 每节主题单一，标题能够独立表达该节意图。
- 关键术语在本节首次出现时补充最小必要定义。
- 跨章节引用使用明确标题或锚点。

#### 12. 抗上下文衰减
- 文档开头提供高密度摘要：文档目标、适用范围、全局约束、关键禁止项、最终产出。
- 全局强约束在对应章节就近重复锚定。
- 长章节提供“本节要点”或“执行清单”。

#### 13. 样例充分
- 抽象规则必须配具体示例：合法值 + 非法值、状态迁移路径、接口输入输出、通过/失败场景。

#### 14. 可验证
- 每条要求都能被检查。
- 验收标准写法：✅ 正常流程、✅ 权限校验、✅ 幂等性、✅ 性能指标。
- 避免无法判断是否达成的表述（如“做好”“优化”）。

### 【维护质量】

#### 15. 版本可追踪
- 文档头部必须包含标准元信息。
- 变更记录必须注明影响范围（示例：v2.1 / 2026-04-10 / 新增幂等键要求 / 影响：所有写接口）。

#### 16. 可复用
- 公共规则可抽离，避免每份文档重复发明。
- 通用约束、术语定义、样例模板可复用。
- 适合同类文档复用同一套结构槽位。

### 【推荐实践】

#### 决策背景记录
- 记录“为什么这样设计”，防止 AI 将关键约束识别为可优化项。

## 四、标准元信息（文档头部模板）

```yaml
---
name: <file-name>
description: <简要描述>
phase: <feasibility|requirements|design|development|testing|release|operations|all>
type: <template|rules|checklist>
stage: <TR1|TR2|TR3|TR4|TR5|all>
status: <draft|stable|deprecated>
version: <x.y.z>
updated: <YYYY-MM-DD>
---
```

## 五、快速检查清单（完成文档后逐一核对）

### ✅ 内容质量
- [ ] 文档开头是否清晰回答了“是什么、给谁用、支撑什么”？
- [ ] 是否明确写出了“做什么”和“不做什么”？
- [ ] 所有关键约束（命名、权限、性能等）是否已显式写出？
- [ ] 是否有明确的优先级标记（必须/应当/可以/暂不要求）？

### ✅ 表达质量
- [ ] 文档结构是否分层清晰、易于扫描？
  - 长文档（>500 行）是否已拆分为 **主入口（TOC）+ 子文档**？
  - 主入口是否包含全局约束和阅读路径？
  - 子文档是否聚焦单一主题、可独立更新？
- [ ] 所有表述是否精确、可量化、无歧义？
- [ ] 术语是否全局一致？（一个概念一个名称）
- [ ] 关键规则是否集中、结论是否前置？
- [ ] 格式是否机器可解析？（标题、列表、表格、代码块）
- [ ] 每节是否自包含、可单独引用？
- [ ] 文档开头是否有高密度摘要？
- [ ] 抽象规则是否配有正例和反例？
- [ ] 每条要求是否都有可验证的验收标准？

### ✅ 维护质量
- [ ] 文档头部是否包含标准元信息？
- [ ] 是否有变更记录，并注明影响范围？
- [ ] 内容是否可复用（公共规则可抽离）？

## 六、反向检查（如果出现以下症状，文档还不够 AI‑Native）

| 症状 | 对应缺失 |
|------|----------|
| 读完不知道要产出什么 | 目标定位（要求 1） |
| 知道目标但不知道怎么判断做对 | 验收标准（要求 14） |
| 正常流程写了，异常靠猜 | 范围与边界（要求 2） |
| 关键规则藏在备注或长段落里 | 上下文自洽（要求 9） |
| 同一概念有多个叫法 | 术语一致（要求 8） |
| 建议和必须无法区分 | 优先级标注（要求 4） |
| 单独摘出一节就不知道适用条件 | 自包含与可引用（要求 11） |
| 长文档里最关键的要求埋在中部，AI 经常漏掉 | 抗上下文衰减（要求 12） |
| 预计或已经超过 500 行仍是单文件长正文 | 结构分层（要求 6） |
| 新旧规则混在同一文档 | 版本可追踪（要求 15） |
| 脱离原作者就失真 | 约束显式化（要求 3） |
| 被引用时语义漂移 | 可复用（要求 16） |

**最终检验标准**：将文档单独交给一个对项目一无所知的 AI，它能否仅凭文档生成出符合预期的产物？如果不能，文档还不够 AI‑Native。

---

## 七、一句话总结

适配 AI‑Native，本质上不是“把文档写得更长”，而是把文档从**说明材料**升级成**协作资产**：

> **既让人能快速理解，又让 AI 能稳定执行，还让工程流程能基于它持续验证、追踪和复用。**
