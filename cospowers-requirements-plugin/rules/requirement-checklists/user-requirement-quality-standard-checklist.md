# TR1 用户需求质量标准（QS 必过项）

> **[AI 读取引导]**
> 本文件是 TR1 用户需求评审的核心质量标准，共 8 条必过项。AI 进行评审时，必须逐条核查每个 QS 项是否在被评审文档中有对应覆盖。
> **评审指令**：
> 1. 对每条 QS 必过项，在被评审文档中查找对应章节
> 2. 核查达成标准是否满足（使用"检查要求"列的量化或可枚举标准）
> 3. 检查是否有模糊词（合理/适当/较好/尽量/性能好/适当考虑）
> 4. 输出每条的"通过/不通过/不涉及（含裁剪说明）"

---

## 高密度摘要

**本文件的目标**：定义 TR1 用户需求阶段的 8 条质量必过项，作为评审和自检的唯一权威来源。

**全局强约束**：
- **必须**：每条必过项在评审时逐条核查，不得跳过
- **必须**：不通过项须与确认人完成备案，方可带风险提交评审
- **禁止**：使用"合理/适当/较好/尽量/性能好/适当考虑"等模糊词描述达成标准
- **禁止**：不通过且未备案的必过项进入评审环节

**8 条 QS 必过项速查表**：

| Rule ID | 检查项标题 | 裁剪原则 | 确认人 |
|---|---|---|---|
| RULE-UR-QS-001 | 完成客户验证（≥3 个） | 不可裁剪 | 产线主管 |
| RULE-UR-QS-002 | 完成协作方需求共识 | 无协作需求可裁剪 | 产线主管 |
| RULE-UR-QS-003 | 严重产品问题纳入需求池 | 不可裁剪 | 蔡泽宜 |
| RULE-UR-QS-004 | 明确 DFX 需求基线 | 不可裁剪 | 蔡泽宜 |
| RULE-UR-QS-005 | 单个需求包含基本内容 | 产线可自定义模板 | 产线主管 |
| RULE-UR-QS-006 | 完成海外需求基线自检 | 不涉及出海可裁剪 | 杜智伟 |
| RULE-UR-QS-007 | 完成道德合规 TR1 检视 | 不可裁剪 | 国内：资质-林芝莉/其他-刘依奇；海外：易泓清 |
| RULE-UR-QS-008 | 包含客户成功相关需求 | 非 XAAS 可裁剪 | 蔡泽宜 |

## RULE-UR-QS-001: 完成客户验证

- **分类**：必过项
- **触发条件**：功能设计完成，提交 TR1 评审前
- **检查要求**：
  1. 已有 ≥3 个客户的验证记录，每条记录包含：客户名称/类型、验证日期、验证场景、客户明确反馈结论（通过/不通过/有条件通过）
  2. 验证对象为"主要价值"（非辅助功能），即围绕核心竞争力的功能
  3. 客户验证结论在文档中可查（非口头）
- **不通过表现**：
  - 验证客户数量 < 3 个
  - 验证记录无具体客户反馈结论，仅有"已验证"等模糊描述
  - 验证对象是辅助功能而非主要价值
  - 无验证记录文档，仅有口头说明
- **裁剪原则**：不可裁剪
- **交付物**：客户验证结论（文档记录或 devops 链接）
- **责任人**：规划经理
- **确认人**：产线主管

## RULE-UR-QS-002: 完成协作方需求共识

- **分类**：必过项
- **触发条件**：文档中存在跨产品线协作需求时
- **检查要求**：
  1. 协作方需求已与协作方规划主管（或对应的规划经理）达成一致
  2. 一致确认结果有记录（邮件/会议纪要/devops 字段），可查
  3. 若无协作需求，需明确写出裁剪声明（而非空白）
- **不通过表现**：
  - 文档存在协作需求章节，但无对方确认记录
  - 仅有"需协作方确认"等待确认状态，未实际完成
  - 有分歧的协作需求被标记为"已共识"但无记录
- **裁剪原则**：不涉及协作需求的可裁剪，但必须在文档中写明"无协作需求，原因：xxx"
- **交付物**：用户需求规格说明书文档（或 devops 必填字段）
- **责任人**：规划经理
- **确认人**：产线主管

## RULE-UR-QS-003: 严重产品问题纳入需求池

- **分类**：必过项
- **触发条件**：提交 TR1 评审前，必须进行问题池检查
- **检查要求**：
  1. 已检索以下三个来源的严重问题：历史遗留库、混沌工程、网上问题
  2. 以下类型的严重问题必须纳入需求池：
     - P0 级别：严重影响/中断客户生产业务（如 HA 失效）
     - P1 级别：业务中断后无法快速恢复
     - 踩中深信服产品质量红线的问题
  3. 纳入的问题在文档"质量需求"章节中有明确记录（问题描述、来源、影响范围、改进方案）
- **不通过表现**：
  - 未检索问题来源，直接声明"无严重问题"
  - 已知 P0/P1 问题未纳入本版本需求池，且无裁剪说明
  - 问题描述过于模糊，无法判断是否为 P0/P1 级别
- **裁剪原则**：不可裁剪（检索行为不可裁剪；若确认无 P0/P1 问题，需明确说明检索过程和结论）
- **交付物**：质量需求（文档中的 P0/P1 问题池章节）
- **责任人**：质量主管
- **确认人**：蔡泽宜

## RULE-UR-QS-004: 明确 DFX 需求基线

- **分类**：必过项
- **触发条件**：所有产品，提交 TR1 评审前。新产品/新平台架构转标准化时要求更严格。
- **检查要求**（三项并列，全部必须满足）：
  1. **全量落地**：已对照 `phases/02-requirements/rules/user-requirement-dfx-baseline-rule.md` 中的 DFX 基线要求逐条检视，输出 DFX 用户需求 Checklist 自检表；不满足项有对应项目 Story 跟踪（Story 链接可查）
  2. **场景 DFX 需求**：针对本版本规划的新增价值场景/特性，已整理用户使用场景需达成的 DFX 能力目标需求列表（可靠性/性能/易用性/合规性/开放性/兼容性/可服务性）
  3. **年度技术规划**：本版本 DFX 需求已包含年度技术规划中计划落地的 DFX 需求内容
- **不通过表现**：
  - 仅有"DFX 需求已考虑"等模糊声明，无 Checklist 自检表
  - 不满足项无 Story 跟踪，仅标注"后续跟进"
  - 新产品未全量对照基线，仅部分检视
  - 年度技术规划中的 DFX 需求未纳入版本需求
- **裁剪原则**：不可裁剪
- **交付物**：DFX 用户需求 Checklist 自检表 + 用户需求文档中的 DFX 需求章节
- **责任人**：产线 DFX 专家
- **确认人**：蔡泽宜

## RULE-UR-QS-005: 单个需求包含基本内容

- **分类**：必过项
- **触发条件**：文档中的每一个单独的用户需求描述
- **检查要求**：每个用户需求应尽可能详细地包含以下内容（不限格式）：
  1. **用户原始诉求**：用户真正想要解决的问题（而非解决方案）
  2. **现状旅程及痛点**：用户当前是如何操作的，痛点是什么
  3. **未来旅程**：结合新功能，用户将如何操作（改善后的旅程）
  4. **主场景描述**：
     - 用户在该场景下如何使用产品（操作步骤）
     - 产品功能需求约束（明确边界）
     - 如有界面，需有界面交互稿（能让评审者理解如何操作）
  5. **次场景描述**：（按实际场景数量填写）
  6. **遗留问题/关键劣势**：客户使用我们方案的缺陷和规避措施
- **不通过表现**：
  - 需求描述仅有功能说明，无用户场景和痛点
  - 无界面交互稿（有界面的功能）
  - 现状旅程与未来旅程描述缺失
  - 遗留问题和劣势未披露
- **裁剪原则**：产线可自定义用户需求模板，但自定义模板需覆盖上述 6 项基本内容；未自定义则按标准模板
- **交付物**：产品包需求（用户需求规格说明书或 devops）
- **责任人**：规划经理
- **确认人**：产线主管

## RULE-UR-QS-006: 完成海外需求基线自检

- **分类**：必过项
- **触发条件**：产品有出海版本时
- **检查要求**：
  1. 已对照下方"海外基线核心分类"逐项自检
  2. 自检结果记录在用户需求文档中，格式包含：基线分类、检视项、自检结果（满足/不满足/不涉及）
  3. 不满足项已纳入版本的产品包需求（有对应需求 ID 或 Story）
  4. 海外需求确认人（杜智伟）已确认
- **海外基线核心分类**：
  - 通用/国际化：语言切换（i18n）、时区支持（全球时区）、夏令时、国际电话号码格式、按地域默认 DNS、多币种、操作系统兼容
  - 系统设计：品牌分离（ForeNova）、证书有效期 398 天、海外软件签名、订阅到期规则与国际竞品对齐
  - 兼容性：Gmail/Outlook 支持、WhatsApp/Line 审计、浏览器兼容、软键盘兼容
  - 配置备份：跨版本配置导入导出、明文配置导出
  - 运维：远程设备管控、设备信息获取（SN/网关 ID）、网络测试工具、在线补丁
  - 安全合规：GPL 合规（Whitesource 扫描）、身份认证/MFA、安全审计日志、敏感数据保护（加密/采集/传输/存储）、个人隐私（GDPR 合规）、防窃听
  - 认证：LDAP/Azure AD/RADIUS、OTP（Google/MS Authenticator）、组织同步、SAML/SSO
- **不通过表现**：
  - 有出海计划但未进行海外基线自检
  - 仅填写"参考海外基线"等空洞声明，无实际自检结果
  - 不满足项未纳入产品包需求
- **裁剪原则**：不涉及出海版本可裁剪，但必须在文档中写明"不涉及出海，原因：xxx"
- **交付物**：海外需求基线（填写满足情况）
- **责任人**：规划经理
- **确认人**：杜智伟

## RULE-UR-QS-007: 完成道德合规 TR1 检视

- **分类**：必过项
- **触发条件**：所有产品，提交 TR1 评审前
- **检查要求**（4 类检视项全部完成，不可跳过任何一类）：
  1. **道德/舆论风险评估**：评估产品是否可能存在道德或舆论风险（涉及个人隐私、侵权/不正当竞争、产品名称合规性）
  2. **重大合规风险评估**：评估产品是否可能存在重大合规风险（渗透/扫描授权要求、禁止性功能、数据跨境合规）
  3. **强制性资质要求评估**：评估是否需要申请新资质或变更现有资质（网专产品目录、商用密码认证等）
  4. **UI 界面合规评估**：评估产品 UI 界面是否符合规范要求（地图显示、品牌信息等）
- **合规检视细化项**：
  - 道德/舆论风险：不得未经授权收集个人数据、不得分发攻击工具、不得预置后门
  - 重大合规风险（国内）：渗透/扫描功能须客户授权、不得实现法律禁止的功能
  - 重大合规风险（海外）：GDPR 合规（欧盟区）、数据本地化、跨境数据传输限制
  - 出口管制：加密算法使用须评估出口许可、吞吐量 >10Gbps 触发许可评估
  - 资质要求（国内）：新增功能涉及资质须提前申请、网专产品目录合规
  - UI 界面要求（国内）：地图使用须符合中国标准地图要求
  - UI 界面要求（海外）：海外版本 UI 须符合目标市场规范
- **不通过表现**：
  - 4 类检视项中有任何一类未完成
  - 检视结果仅填"通过"但无说明/举证
  - 存在风险但未联系对应法务接口人确认
  - 海外产品未完成海外特殊合规评估
- **裁剪原则**：不可裁剪
- **交付物**：道德及合规遵从 Checklist — TR1 阶段（按上述 4 类检视项逐条填写自检结果）
- **责任人**：规划经理/规划主管
- **确认人**：
  - 国内资质问题：林芝莉；其他国内合规：刘依奇
  - 海外合规：易泓清

## RULE-UR-QS-008: 包含客户成功相关需求

- **分类**：必过项
- **触发条件**：XAAS 类业务产品
- **检查要求**（3 项全部满足）：
  1. **新增/改动功能已考虑客户成功需求**：功能设计已考虑客户实现成功所需的功能支撑（不仅是功能本身）
  2. **必备平台需求**：已包含必备的客户成功平台相关需求
  3. **基线自检完成**：已完成 XAAS 客户成功基线需求自检，不通过及不涉及项已与确认人（蔡泽宜）确认，并附上确认结果
- **不通过表现**：
  - XAAS 产品未包含任何客户成功需求章节
  - XAAS 客户成功基线需求自检未完成
  - 有不通过项但未与蔡泽宜确认
- **裁剪原则**：非 XAAS 类业务可裁剪，但必须在文档中写明"非 XAAS 类业务，原因：xxx"
- **交付物**：XAAS 客户成功基线需求 Checklist
- **责任人**：规划经理
- **确认人**：蔡泽宜

| 步骤 | 活动名称 | 活动目的 | 负责人 |
